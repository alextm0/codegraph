    Graph-Based Context Selection for AI Coding Agents

    Overview

    This project builds a tool that helps AI coding assistants (Claude Code, Cursor, GitHub Copilot) find the right code context when working on a task.

    Instead of searching for textually similar code (keyword matching, embeddings), this system understands how code is structurally connected — through function calls, imports, and inheritance — and uses those connections to find what's relevant.

    In one sentence: Parse a codebase into a graph of dependencies, use Personalized PageRank to find structurally relevant code given a task, and expose this through the Model Context Protocol (MCP) so any AI agent can use it.

    The Core Insight

    Code is a graph, not text.

    • Nodes = code entities: functions, classes, methods, files
    • Edges = relationships: function A calls function B, file X imports file Y, class C inherits from class D

    Graph-based search follows structural connections to find relevant code regardless of textual similarity. A function BaseHasher in a distant file with different variable names is still found if it's connected to authenticate() through the call chain.

    System Pipeline

    The system has four stages, each feeding into the next:

    STAGE 1: PARSE
  Input:  A Python repository (source code files)
  Action: Parse each file into an AST using tree-sitter
  Output: Syntax tree per file
    │
    ▼
STAGE 2: EXTRACT & RESOLVE
  Input:  Syntax trees from Stage 1
  Action: Extract entities (functions, classes, imports),
          resolve cross-file connections (imports → files, calls → definitions)
  Output: Lists of nodes (entities) and edges (relationships)
    │
    ▼
STAGE 3: BUILD GRAPH & RUN PPR
  Input:  Nodes and edges from Stage 2
  Action: Load into Neo4j, create in-memory projection,
          run Personalized PageRank from seed nodes
  Output: Ranked list of code entities with relevance scores
    │
    ▼
STAGE 4: SERVE
  Input:  The graph database + PPR algorithm (on-demand)
  Action: Expose as an MCP server that AI agents can call
  Output: Top-k most relevant code snippets with source

    Performance Expectations

    Operation Expected Time
    tree-sitter parsing (100k LOC) 0.5–2 seconds
    Entity extraction + resolution 1–5 seconds
    Neo4j graph loading (batched) 2–15 seconds
    GDS projection creation < 1 second
    PPR execution 1–10 milliseconds
    Source code reading + formatting 50–200 milliseconds
    Full pipeline (parse → result) 5–30 seconds
    Query only (graph already built) < 250 milliseconds

    The graph is built once per repository state. Subsequent queries take under 250ms.

    Known Limitations

    1. Static analysis only — Dynamic behaviors (monkey-patching, eval()) are invisible.
    2. Python only — Architecture is language-extensible, but this thesis covers Python.
    3. Imperfect import resolution — Custom resolver handles ~80% of imports; complex patterns may be missed.
    4. Imperfect call resolution — Without type inference, duck-typed method calls are unresolvable.
    5. Utility function noise — IDF weighting reduces but may not eliminate highly-connected generic functions.
    6. No incremental updates — Graph must be rebuilt when code changes (5–30 seconds).
    7. PPR may not always win — BM25 or 1-hop ego-graphs may perform comparably for some tasks.
