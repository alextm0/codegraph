    Post-Processing

    PPR returns a ranked list of nodes with scores. Before sending results to the AI agent, two post-processing steps improve quality.

    Problem 1: The Utility Function Problem

    What Happens

    A function like log_error() is called by 200 other functions across the codebase. Because it's connected to everything, PPR from any seed will rank it highly. But log_error() is not useful context — it's generic infrastructure code.

    Analogy: In Google Search, if "Wikipedia" appeared as #1 for every query, that's technically correct (Wikipedia is linked everywhere) but not helpful.

    Solution: IDF-Inspired Edge Weight Adjustment

    IDF (Inverse Document Frequency) is a concept from text search: terms that appear in many documents are less informative than terms that appear in few. "The" appears everywhere → low weight. "Authenticate" appears rarely → high weight.

    Applied to our graph: functions called by many others are less informative than functions called by few. We adjust edge weights:

    adjusted_weight(A → B) = base_weight(edge_type) / log₂(in_degree(B) + 2)

    Where in_degree(B) = number of incoming edges to node B.

    The Effect

    Function in_degree Weight Adjustment Effect
    log_error() 200 base / log₂(202) ≈ base / 7.7 Heavily downweighted
    hash_password() 3 base / log₂(5) ≈ base / 2.3 Slightly downweighted
    calculate_token_expiry() 1 base / log₂(3) ≈ base / 1.6 Barely affected

    PPR naturally reduces utility function scores while preserving scores for specific, task-relevant functions.

    Base Weights by Edge Type

    Edge Type Base Weight Rationale
    INHERITS_FROM 1.0 Strongest coupling — base class changes affect all subclasses
    CALLS 0.8 Direct dependency — callee changes may break caller
    IMPORTS 0.5 File-level connection — weaker than function-level
    CONTAINS 0.4 Structural containment — relevant but not a dependency

    These weights are applied when creating the GDS graph projection and used by PPR during the random walk.

    Problem 2: Token Budgeting

    The Challenge

    PPR returns node names with scores. The AI agent needs actual source code, not just names. But we can't send unlimited code — there's a token budget.

    The Process

    1. Take the top-k results from PPR (e.g., top 20)
    2. For each result, read the actual source code using stored file_path, line_number, end_line
    3. Accumulate results until reaching the token budget
    4. Format and return
    def format_context(ppr_results, token_budget=6000):
    context_parts = []
    tokens_used = 0

    for node, score in ppr_results:
        source_code = read_lines(node.file_path, node.line_number, node.end_line)
        source_tokens = count_tokens(source_code)

        if tokens_used + source_tokens > token_budget:
            break  # Budget filled

        context_parts.append({
            "entity": node.name,
            "file": node.file_path,
            "lines": f"{node.line_number}-{node.end_line}",
            "relevance_score": round(score, 4),
            "source_code": source_code
        })
        tokens_used += source_tokens

    return context_parts

    Why 4,000–8,000 Tokens

    The AI agent has its own context window to manage (100K–200K tokens). Our context is one of many inputs. 4,000–8,000 tokens of highly relevant code is more useful than 50,000 tokens of loosely relevant code — the agent can focus on what matters.
