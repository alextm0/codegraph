    tree-sitter & Parsing

    This document covers two tightly coupled steps: parsing source code into syntax trees (tree-sitter) and extracting entities from those trees.

    What tree-sitter Is

    tree-sitter is a parser library. A parser reads source code (plain text) and converts it into a structured tree representing the code's syntax. This tree is called an Abstract Syntax Tree (AST).

    Analogy: Parsing is like diagramming a sentence in English class. "The dog chased the cat" becomes: subject ("The dog"), verb ("chased"), object ("the cat"). An AST does the same for code: this is a function definition, that's a function call, this is an if-statement.

    What tree-sitter Produces

    Given this Python code:

    def authenticate(username, password):
    user = find_user(username)
    if user and check_password(user, password):
        return create_token(user)
    return None

    tree-sitter produces a tree like:

    function_definition
  name: "authenticate"
  parameters: ["username", "password"]
  body:
    assignment
      left: "user"
      right: call(name: "find_user", args: ["username"])
    if_statement
      condition: ...
      body:
        return_statement
          value: call(name: "create_token", args: ["user"])
    return_statement
      value: None

    This tells us: there's a function called authenticate, and it calls find_user, check_password, and create_token.

    Why tree-sitter

    Property Detail
    Fast ~5–10ms per 1,000 lines. A 100,000-line codebase takes under 2 seconds.
    Multi-language Supports 100+ languages (Python, JS, TS, Java, Go, Rust…).
    Industry standard Used by GitHub, Cursor, and many other tools.
    Error-tolerant Handles broken code — marks broken parts and keeps going.

    What tree-sitter Cannot Do

    This is critical:

    • Cannot resolve imports — Sees from utils import helper as text, doesn't know which file utils refers to.
    • Cannot resolve types — Sees obj.method() but doesn't know which class obj is.
    • Cannot look across files — Only parses one file at a time.

    tree-sitter gives us the raw material. Connecting entities across files is handled by Import & Call Resolution.

    How tree-sitter Is Used in This Project

    import tree_sitter_python as tspython
from tree_sitter import Language, Parser

parser = Parser(Language(tspython.language()))

with open(file_path, 'rb') as f:
    source = f.read()

tree = parser.parse(source)
# tree.root_node gives access to the entire AST

    tree-sitter also provides a query language for pattern matching:

    (function_definition name: (identifier) @function.name)

    This says: "find every function_definition node and capture its name child." The @function.name part is like naming a capture group in regex.

    Entity Extraction

    After tree-sitter produces a syntax tree per file, we walk through it and pull out the entities that become nodes in our graph.

    What We Extract

    Entity Example Captured Properties
    Function (standalone) def validate_email(email): name, file_path, line_number, end_line, signature, docstring
    Class class AuthService(BaseService): name, file_path, line_number, end_line, bases (parent classes)
    Method (inside a class) class AuthService: → def authenticate(self, ...): Same as Function + class_name
    Import from src.utils.helpers import validate_email module_path, imported_names, is_relative
    Function call (inside bodies) user = find_user(username) caller_name, callee_name, line_number

    What We Skip (and Why)

    Skipped Reason
    Standard library imports (import json) Part of Python, not the project — adds noise without structural value.
    Third-party library calls (requests.get()) Same — we only care about project-internal connections.
    Variables and constants Not included as nodes for simplicity. Functions and classes capture the meaningful relationships.
    Decorators, type annotations, comments Not extracted as separate entities (decorators noted as properties).

    Extraction Output

    For each file, the extractor produces:

    {
    "file": "src/services/auth_service.py",
    "classes": [
        {"name": "AuthService", "line": 10, "end_line": 85, "bases": ["BaseService"]},
    ],
    "functions": [],
    "methods": [
        {"name": "authenticate", "class": "AuthService", "line": 12, "end_line": 30,
         "signature": "(self, username: str, password: str) -> bool"},
        {"name": "refresh_token", "class": "AuthService", "line": 32, "end_line": 50,
         "signature": "(self, token: str) -> str"},
    ],
    "imports": [
        {"module": "src.models.user", "names": ["User"], "is_relative": False},
        {"module": "src.utils.crypto", "names": ["hash_password"], "is_relative": False},
    ],
    "calls": [
        {"caller": "AuthService.authenticate", "callee": "hash_password", "line": 22},
        {"caller": "AuthService.authenticate", "callee": "User.find_by_username", "line": 18},
    ],
}

    After processing all files, the raw data is passed to the resolution step.

    Qualified Names

    To uniquely identify entities across the codebase, we use qualified names — the full path to an entity:

    Entity Qualified Name
    Function validate_email in src/utils/helpers.py src.utils.helpers.validate_email
    Method authenticate on AuthService in src/services/auth_service.py src.services.auth_service.AuthService.authenticate
    Class User in src/models/user.py src.models.user.User

    This prevents ambiguity when multiple files have functions with the same name.
