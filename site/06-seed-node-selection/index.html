    Seed Node Selection

    PPR starts from seed nodes and spreads outward. The quality of the seeds directly determines the quality of the results. Good seeds → relevant context. Bad seeds → useless context.

    The Core Question

    How does the system decide where to start PPR?

    Why "Current File" Is a Bad Default

    A naïve approach: "the developer is editing auth_service.py, so use all functions in that file as seeds." This fails because:

    • The developer might have been searching for an hour and landed on a random file
    • The developer might be reading code, not editing the part that matters
    • The AI agent might be exploring multiple files in a multi-step task

    "Where the cursor is" ≠ "what the developer cares about."

    The Actual Approach: Agent-Provided Context

    The MCP server doesn't guess the seed — the AI agent calling the tool tells it what the task is. When a developer says "fix the authentication timeout bug" to Claude Code, the agent already understands the task. It passes that understanding to our tool via the tool parameters.

    @mcp.tool()
def get_relevant_context(
    task_description: str,                  # "fix the authentication timeout bug"
    mentioned_entities: list[str] = None,   # ["authenticate", "TokenExpiredError"]
    current_file: str = None                # optional low-weight hint
) -> str:

    The Seed Extraction Pipeline

    Seeds come from multiple signals with different confidence levels:

    Step 1: Explicit Entity Extraction (HIGH weight)

    Match explicitly mentioned names against graph nodes.

    Input:  task_description = "fix the authentication timeout in AuthService"
        mentioned_entities = ["authenticate", "AuthService", "TokenExpiredError"]
Action: Match these names against nodes in the graph
Result: AuthService (exact match), authenticate method (exact match)

    Step 2: BM25 Keyword Search (MEDIUM weight)

    Search the task description text against all function signatures and docstrings.

    Input:  task_description text
Action: BM25 ranks functions by textual relevance to the task
Result: Top 3–5 functions mentioning "timeout", "token", "auth", etc.

    What is BM25? A text search algorithm. Given a query ("authentication timeout") and a set of documents (all function signatures + docstrings), it ranks documents by how well they match. It considers: does the document contain the query words? How many times? How long is the document? BM25 is the standard baseline in information retrieval — simple, fast, and effective.

    Step 3: Current File as Weak Hint (LOW weight)

    If provided, add functions from the current file as seeds.

    Input:  current_file = "src/services/auth_service.py"
Action: Add functions from this file as seeds

    Step 4: Build the Personalization Vector

    Combine all seeds with their weights, normalize to sum to 1.0:

    {
  AuthService.authenticate: 0.35,
  AuthService: 0.25,
  timeout_handler: 0.15,
  token_validator: 0.10,
  AuthService.refresh_token: 0.08,
  AuthService.logout: 0.07
}

    This becomes the personalization vector for PPR.

    Why Multi-Signal Matters

    Different scenarios benefit from different signals:

    Scenario Best Signal Why
    "Fix bug in authenticate()" Explicit entity extraction Clear function name mentioned
    "The login flow is broken" BM25 keyword search No specific function, but keywords help
    "Continue working on the auth module" Current file + BM25 Vague task, file hint narrows scope
    SWE-bench evaluation (bug report text) Entity extraction + BM25 Bug reports often mention specific classes

    By combining signals with weights, the system degrades gracefully — even when one signal is weak, the others compensate.
