    Personalized PageRank (PPR)

    This is the core algorithm of the project. This document explains it from scratch — no prior knowledge assumed.

    Starting with Regular PageRank

    PageRank was invented by Larry Page and Sergey Brin (Google founders, 1998) to rank web pages.

    The intuition: Imagine a person randomly browsing the web. They start on a random page, randomly click a link, then another, and so on. Pages they visit most often are the most "important."

    Why? Important pages have many incoming links from other important pages. The random surfer ends up at these pages frequently.

    The teleportation trick: What if the surfer reaches a dead end or a loop? With some probability (typically 15%), the surfer "teleports" — jumps to a completely random page. This ensures they eventually visit every page.

    The result: a score for every page representing visit frequency. Higher score = more important.

    What Makes PPR Different

    In regular PageRank, teleportation goes to any random page with equal probability → global importance ranking.

    In Personalized PageRank (PPR), teleportation goes back to specific pages you choose — called seed nodes. This biases the walk toward a particular region of the graph.

    Algorithm Teleports to Answers
    PageRank Any random node "What are the most important nodes globally?"
    PPR Specific seed nodes "What are the most important nodes relative to my seeds?"

    For code: PageRank tells you the most important functions in the entire codebase. PPR from authenticate() tells you the most important functions relative to authenticate() — the ones most structurally connected to it.

    How PPR Works Step by Step

    Given a graph and a seed node (e.g., authenticate()):

    1. Start at the seed. Place all probability mass (1.0 units of "paint") on the seed.

    2. Walk step. From each node with paint, spread it along edges. If a node has 3 edges, each gets 1/3 of the paint. But only spread (1 − α) of the paint — keep α at the seed.

    3. α (alpha) = restart probability (typically 0.15). Controls how far paint spreads. Higher α = stays closer to seed.

    4. Repeat ~20 iterations. The paint distribution converges.

    5. Read scores. The paint at each node = its PPR score = its relevance to the seed.

    Concrete Example

    Seed: authenticate()

After PPR:
  authenticate()          → 0.150  (high — it's the seed)
  hash_password()         → 0.082  (high — directly called by seed)
  check_token()           → 0.067  (high — directly called by seed)
  User.find_by_name()     → 0.045  (medium — called by seed)
  BaseService.init()      → 0.023  (medium — inherited by seed's class)
  process_login()         → 0.018  (medium — calls the seed)
  log_error()             → 0.012  (lower — connected but generic)
  send_email()            → 0.001  (very low — barely connected)

    Key Parameters

    Parameter Controls Default Effect of Increasing
    α (alpha) / restart probability How close results stay to seed 0.15 Results cluster tighter around seed
    maxIterations Precision of scores 20 More precise (diminishing returns after ~20)
    tolerance Early stopping threshold 1e-7 Lower = more precise but slower
    top-k Number of results returned 10–20 More context but more noise / tokens used

    In Neo4j GDS, dampingFactor = 1 − α. So dampingFactor = 0.85 means α = 0.15 (15% teleport chance).

    Why PPR Is the Right Algorithm

    Approach How It Works Limitation
    1-hop neighbors Return everything directly connected Misses indirect connections (A→B→C — C is missed)
    k-hop ego-graph (RepoGraph) Return everything within k edges k=1 too narrow; k=2 explodes; no ranking within results
    Shortest path Find closest entities by edge count Doesn't account for multiple paths or connection strength
    PPR (this project) Probabilistic relevance via all paths Slightly more complex — but Neo4j GDS handles it

    PPR's Advantages

    • Multi-hop: Finds indirectly connected but important code
    • Ranked: Every node gets a score → take exactly top-k
    • Multiple paths matter: Node X connected through 5 paths scores higher than Y through 1 path
    • Mathematically principled: Guaranteed convergence, provably correct
    • Fast: 1–10ms on 50,000-node graphs

    Directionality

    Edges have a natural direction. authenticate() CALLS hash_password() is an edge from authenticate to hash_password. PPR can follow edges in different directions:

    Direction What PPR Finds Example
    Natural (follow edge direction) Things the seed depends on (downstream) authenticate → finds hash_password
    Reverse (follow edges backward) Things that depend on the seed (upstream) authenticate → finds process_login
    Undirected (both ways) Everything structurally connected Finds both hash_password AND process_login

    Decision: UNDIRECTED as default. An AI agent fixing authenticate() needs to see hash_password() (to understand the implementation) AND process_login() (to check if the fix affects callers).

    Evaluation includes directed and reversed PPR as ablation experiments to measure which direction works best.

    Running PPR in Neo4j

    -- Find the seed node
MATCH (seed:Function {name: 'authenticate', file_path: 'src/services/auth_service.py'})

-- Run PPR from that seed
CALL gds.pageRank.stream('codeGraph', {
  maxIterations: 20,
  dampingFactor: 0.85,
  sourceNodes: [seed],
  relationshipWeightProperty: 'weight'
})
YIELD nodeId, score

-- Return top 20 results
RETURN gds.util.asNode(nodeId).name AS name,
       gds.util.asNode(nodeId).file_path AS file,
       score
ORDER BY score DESC
LIMIT 20

    Multiple Seeds

    PPR can accept multiple seed nodes with different weights:

    MATCH (a:Function {name: 'authenticate'}), (b:Function {name: 'refresh_token'})
CALL gds.pageRank.stream('codeGraph', {
  maxIterations: 20,
  dampingFactor: 0.85,
  sourceNodes: [a, b]
})

    Paint is distributed equally across seeds. GDS also supports weighted seeds to give more importance to one over another.
