    Evaluation

    This document covers how the system is tested: the benchmark, the metrics, the baselines, and the ablation studies.

    What We're Measuring

    The fundamental question: Does PPR retrieve the right code?

    We answer this with real bug reports where we know which files were actually changed to fix the bug. If PPR's top results include those files, PPR "got it right."

    The Benchmark: SWE-bench Lite

    SWE-bench Lite is a dataset of 300 real bug reports from popular Python repositories (Django, Flask, scikit-learn, matplotlib, etc.).

    For each bug, the dataset provides:

    Component Description
    Repository state before the fix A specific git commit
    Bug description The GitHub issue text
    Gold patch The actual code changes — which files and functions were changed

    We don't fix bugs or run an LLM. We just ask: given this bug description, does PPR point to the files that were actually changed?

    Evaluation Process

    For each of the 300 bug reports:
  1. Check out the repository at the "before fix" commit
  2. Build the code graph (tree-sitter → Neo4j)
  3. Extract seed nodes from the bug description text
  4. Run PPR from the seeds, get top-k results
  5. Check: do the top-k results include the files from the gold patch?
  6. Record hit/miss and rank of first correct result

    Metrics

    Recall@k

    "Out of all files that needed changing, what percentage appeared in the top k results?"

    Example: Bug required changing 3 files. PPR's top 10 included 2 of them → Recall@10 = 66.7%

    We report Recall@5 and Recall@10.

    MRR (Mean Reciprocal Rank)

    "On average, where did the first correct file appear in the ranking?"

    First correct at #1 → 1.0 | at #3 → 0.33 | at #7 → 0.14

    MRR = average across all 300 bug reports. Higher = correct files appear earlier.

    Baselines

    Baseline How It Works Why Include It
    Random Pick k random files Floor — any method must beat this
    BM25 Keyword search against code contents Standard text-retrieval baseline
    1-hop ego-graph Return direct graph neighbors (no PPR) Tests whether multi-hop adds value

    Ablation Studies

    An ablation study removes or changes one component to measure its contribution.

    Experiment What Changes What It Reveals
    Remove CALLS edges Only IMPORTS, INHERITS, CONTAINS Importance of call relationships
    Remove IMPORTS edges Only CALLS, INHERITS, CONTAINS Importance of imports
    Remove INHERITS edges Only CALLS, IMPORTS, CONTAINS Importance of inheritance
    Change α α ∈ Optimal PPR spread distance
    Change direction Directed vs reversed vs undirected Best edge direction
    IDF weighting on/off Utility function handling toggle Value of downweighting common functions
    PPR vs 1-hop PPR vs simple neighbor lookup Algorithm sophistication value
    Change top-k k ∈ Results needed for good recall

    Each ablation is a separate run with one variable changed — valuable thesis content regardless of outcome.
