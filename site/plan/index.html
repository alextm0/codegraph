    CodeGraph: Implementation Plan for Remaining Stages

    Context

    What's done: Stages 1-3 (parsing, resolution, Neo4j graph, PPR) are fully implemented and tested (144 tests). The system can parse Python repos, build dependency graphs in Neo4j, and run Personalized PageRank.

    What's missing: The "brain" that connects the task description to the graph (seed selection), the quality refinements (IDF + token budgeting), and the interface layer (MCP server). Without these, the system has all the components but no way for an AI agent to use them end-to-end.

    Goal: Implement stages 4-5 so the full pipeline works: task description -> seed extraction -> PPR -> post-processing -> formatted source code exposed via MCP.

    Stage 4a: Seed Node Selection (src/retrieval/seed_selection.py)

    The thinking

    PPR needs starting nodes ("seeds"). The quality of seeds determines the quality of results. The question is: given a task like "fix the authentication timeout bug", how do we pick the right graph nodes to start from?

    Three signals, ranked by confidence:

    1. Explicit entity matching (HIGH) — If the task mentions "AuthService" or "authenticate", match those directly against node names in the graph. This is the most reliable signal because the agent explicitly told us what matters.

    2. BM25 keyword search (MEDIUM) — Search the task text against function signatures and docstrings stored in Neo4j. BM25 is a standard text-ranking algorithm (like a smarter ctrl+F). It finds functions whose descriptions are textually similar to the task, even when no exact entity name is mentioned.

    3. Current file hint (LOW) — If the agent says "I'm currently in auth_service.py", add entities from that file as weak seeds. Low weight because "where the cursor is" does not always equal "what matters".

    These signals are combined into a personalization vector — a dictionary mapping node IDs to weights that sum to 1.0 — which PPR uses as its teleportation distribution.

    New module: src/retrieval/seed_selection.py

    Why a new retrieval/ package? Seed selection and post-processing are about retrieving relevant context from the graph. They sit between the graph layer (storage/algorithms) and the MCP layer (interface). Grouping them in src/retrieval/ keeps each module single-purpose per CLAUDE.md guidelines.

    Data model:

    @dataclass(frozen=True)
class SeedNode:
    """A single seed with its source signal and raw weight."""
    node_id: int
    qualified_name: str
    weight: float
    source: str  # "entity_match", "bm25", "current_file"

@dataclass(frozen=True)
class PersonalizationVector:
    """Normalized seed weights ready for PPR."""
    seeds: dict[int, float]  # node_id -> normalized weight (sums to 1.0)

    Functions:

    extract_seeds(
    driver, task_description, mentioned_entities, current_file,
    signal_weights
) -> PersonalizationVector

_match_entities(driver, mentioned_entities) -> list[SeedNode]
_bm25_search(driver, task_description, top_n) -> list[SeedNode]
_current_file_seeds(driver, current_file) -> list[SeedNode]
_normalize_seeds(all_seeds) -> PersonalizationVector

    Dependencies to add: rank-bm25 (already listed in CLAUDE.md tech stack but not in pyproject.toml yet).

    How BM25 works here: We fetch all Function/Method nodes with their signature and docstring from Neo4j, build a BM25 index in memory, query it with the task description tokens, and take the top N results. This is fast (< 50ms for 10k nodes) and doesn't require any external service.

    Config additions to config.yaml:

    seed_selection:
  entity_match_weight: 0.6    # weight multiplier for exact entity matches
  bm25_weight: 0.3            # weight multiplier for BM25 results
  current_file_weight: 0.1    # weight multiplier for current file entities
  bm25_top_n: 5               # how many BM25 results to include

    Stage 4b: Post-Processing (src/retrieval/post_processing.py)

    The thinking

    PPR returns raw scores, but two problems remain:

    Problem 1: Utility functions bubble up. A function like log_error() called by 200 others appears highly ranked for any seed because it has so many incoming edges. It's structurally important but not task-relevant.

    Solution: IDF-inspired edge reweighting. Before running PPR, adjust edge weights based on the target node's in-degree:

    adjusted_weight = base_weight / log2(in_degree + 2)

    The +2 prevents division by zero and keeps single-connection nodes nearly unchanged. Functions with many callers get heavily downweighted. This is applied when creating the GDS projection — we update the weight property on edges in Neo4j before projecting.

    Problem 2: Token budget. PPR returns node names and scores, but the AI agent needs actual source code. We can't send everything — we need to stay within a token budget (4,000-8,000 tokens).

    Solution: Walk the ranked results top-down, read source code for each, accumulate until the budget is reached.

    New module: src/retrieval/post_processing.py

    Functions:

    apply_idf_weights(driver) -> int
    # Updates edge weights in Neo4j based on target node in-degree.
    # Returns the number of edges updated.
    # Must be called BEFORE project_graph() in the pipeline.

format_context(
    ppr_results, project_root, token_budget

    # Reads source code for each PPR result, accumulates until budget.

count_tokens(text) -> int

    Data model:

    @dataclass(frozen=True)
class ContextResult:
    """A single code entity with its source code, ready to serve."""
    entity_name: str
    qualified_name: str
    file_path: str
    line_start: int
    line_end: int
    relevance_score: float
    source_code: str
    token_count: int

    Why whitespace tokenization is fine

    The doc suggests 4,000-8,000 tokens. We're using this as a rough budget, not billing. len(text.split()) correlates ~0.75x with actual LLM tokens. Close enough for budgeting — and zero dependencies.

    Test file: tests/test_post_processing.py

    • Test IDF: high in-degree nodes get lower edge weights
    • Test token counting: basic correctness
    • Test format_context: respects budget, returns results in score order
    • Test edge case: budget smaller than first result still returns one result
    • IDF tests require Neo4j; token counting/formatting tests are pure functions (no Neo4j)

    Stage 5: MCP Server (src/mcp_server.py)

    The thinking

    MCP (Model Context Protocol) is how AI agents discover and call tools. Our server exposes three tools:

    1. get_relevant_context — The main tool. Full pipeline: seeds -> PPR -> post-processing -> source code. This is what the agent calls when it needs to understand code relevant to a task.

    2. query_dependencies — A simpler tool. Given an entity name, return its direct dependencies (what it calls, what calls it). No PPR — just raw graph traversal. Useful for focused exploration.

    3. get_graph_stats — Diagnostic. Returns node counts, edge counts, most connected files. Helps the agent understand the codebase shape.

    Architecture

    The MCP server is a single-file entry point (src/mcp_server.py) that: 1. On startup: loads config, connects to Neo4j, verifies connectivity 2. On first get_relevant_context call: applies IDF weights and creates GDS projection (lazy, since the graph may not be built yet) 3. On tool call: runs the pipeline and returns formatted JSON

    Why single file? The MCP server is thin glue — it calls into retrieval/ and graph/. Each tool handler is ~15-20 lines. No need for a package.

    Tool signatures:

    @mcp.tool()
async def get_relevant_context(
    task_description: str,
    mentioned_entities: list[str] | None = None,
    current_file: str | None = None,
    top_k: int = 15,
    token_budget: int = 6000,
) -> str:
    """Find structurally relevant code for a task using graph-based ranking."""

@mcp.tool()
async def query_dependencies(
    entity_name: str,
    direction: str = "both",
    depth: int = 1,
) -> str:
    """Get direct dependencies for a code entity."""

@mcp.tool()
async def get_graph_stats() -> str:
    """Return statistics about the code dependency graph."""

    Pipeline Integration: src/retrieval/pipeline.py

    The thinking

    The MCP server shouldn't contain pipeline logic. A separate pipeline.py orchestrates the full flow so it can be tested independently of MCP.

    def run_retrieval_pipeline(
    driver, gds, task_description, mentioned_entities,
    current_file, ppr_config, token_budget, project_root,
) -> list[ContextResult]:
    """Full pipeline: seed extraction -> PPR -> post-processing."""
    seeds = extract_seeds(driver, task_description, mentioned_entities, current_file)
    ppr_results = run_ppr_from_node_ids(gds, driver, list(seeds.seeds.keys()), ppr_config)
    return format_context(ppr_results, project_root, token_budget)

    This function is what the MCP tool calls. It's also what the evaluation harness will call.

    Test file: tests/test_pipeline.py

    • End-to-end test: task description -> formatted source code results
    • Uses user_auth fixture

    What's NOT in this plan

    • Evaluation (Stage 6) — SWE-bench benchmarking. This is a separate effort that should come after the pipeline is stable. It involves cloning repos, running the pipeline on 300 bug reports, computing metrics. Worth its own plan.
    • REST API fallback — Mentioned in docs but not essential. MCP is the primary interface.
    • CLI entry point — A python -m codegraph command for building graphs from CLI. Nice to have but not blocking.
