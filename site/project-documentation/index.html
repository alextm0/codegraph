    Graph-Based Context Selection for AI Coding Agents

    Project Reference Documentation

    This document explains every part of the thesis project from the ground up. It is written to be a reference you can come back to at any point during development. Every technical term is explained when first introduced. Every design decision includes the reasoning behind it.

    1. What This Project Is

    This project builds a tool that helps AI coding assistants (like Claude Code, Cursor, GitHub Copilot) find the right code context when working on a task.

    When an AI assistant needs to fix a bug or add a feature, it needs to "see" the relevant parts of the codebase. Today's tools find relevant code by searching for textually similar code — like googling for keywords. This project replaces that with structural search — understanding how code is actually connected through function calls, imports, and inheritance, then using those connections to find what's relevant.

    The output is an MCP server — a standardized plugin that any AI coding agent can connect to and ask: "Given this task, what code should I look at?"

    In one sentence: Parse a codebase into a graph of dependencies, use a graph algorithm (Personalized PageRank) to find structurally relevant code given a task, and expose this through a standard protocol any AI agent can use.

    2. The Problem: How AI Coding Tools Pick Context Today

    What "context" means

    When an AI assistant works on your code, it can't see your entire codebase at once. It has a limited window (called a context window — the amount of text the AI can process at once, typically 100,000-200,000 tokens). A medium-sized project might have 500,000+ tokens of code. So the AI has to choose: which parts of the codebase are worth "seeing" for this particular task?

    This selection process is called context selection or context retrieval. The quality of the AI's output depends heavily on whether it sees the right code.

    How current tools do it

    Cursor and GitHub Copilot use a technique called embedding-based retrieval. Here's how it works in plain terms:

    1. Every chunk of code gets converted into a list of numbers (a "vector" or "embedding") that represents its meaning. Think of it like a fingerprint for each piece of code.
    2. When you ask a question or edit code, your question also gets converted into a similar list of numbers.
    3. The tool finds code chunks whose number-lists are most similar to your question's number-list.
    4. Those "similar" chunks become the context the AI sees.

    This is essentially keyword search on steroids — it finds code that talks about similar things as your question.

    Claude Code takes a different approach: it doesn't pre-build an index at all. Instead, it uses tools (like grep and file reading) to search the codebase on-the-fly, deciding what to look at as it goes. This avoids stale indexes but relies on the AI making good search decisions.

    Why this approach fails for code

    Code is not like a book or a Wikipedia article. Consider this scenario:

    You need to fix a bug in authenticate(). This function calls hash_password(), which uses BCryptHasher, which inherits from BaseHasher. The BaseHasher class is defined in a completely different file with completely different variable names and completely different comments.

    Embedding-based search will miss BaseHasher because it looks nothing like authenticate() textually. But structurally, it's critically important — if you change how hashing works, BaseHasher is part of the chain.

    The fundamental issue: embeddings find code that looks similar in text, but code relevance is determined by structural connections — what calls what, what imports what, what inherits from what. These connections form a graph, and a graph requires graph-based search, not text-based search.

    A Google DeepMind paper from August 2025 formally proved that embedding-based retrieval has mathematical limitations: there exist relevant documents that cannot be retrieved by any query in an embedding space. This is a theoretical ceiling, not just a practical one.

    Five major papers from top conferences in 2024-2025 (NAACL, ICLR, ACL, ASE) independently showed that using code's graph structure for retrieval substantially outperforms text-based approaches, with improvements ranging from +3% to +33%.

    3. The Core Idea: Code Is a Graph, Not Text

    What "code is a graph" means

    A graph is a data structure with two things: nodes (things) and edges (connections between things). Think of a social network: people are nodes, friendships are edges.

    Code has the same structure:

    • Nodes are code entities: functions, classes, files, methods
    • Edges are relationships between them: function A calls function B, file X imports file Y, class C inherits from class D

    For example, in a web application:

      [app.py] ---IMPORTS---> [auth_service.py] ---IMPORTS---> [user_model.py]
      |                         |                              |
  has function             has class                      has class
      |                         |                              |
  [main()]               [AuthService]                    [User]
      |                     |       |                         |
   CALLS              has method   has method             INHERITS_FROM
      |                  |              |                      |
      +--------> [authenticate()]  [refresh_token()]     [BaseModel]
                       |
                    CALLS
                       |
                 [hash_password()]

    This is a code dependency graph. Every codebase has one, whether or not anyone has explicitly built it. The relationships exist in the source code — you just have to extract them.

    Why graph search is better than text search for code

    When you ask "what code is relevant to fixing a bug in authenticate()?", a graph can follow the connections:

    • authenticate() calls hash_password() → probably relevant
    • authenticate() is in AuthService which inherits from BaseService → might need to check
    • process_login() calls authenticate() → will be affected by your change
    • user_model.py is imported by auth_service.py → the User class is probably involved

    Text search would only find code that contains words like "authenticate", "login", "password". The graph finds code that is structurally connected regardless of what words it uses.

    The specific algorithm: Personalized PageRank

    To rank which connected nodes are most relevant, this project uses Personalized PageRank (PPR). Full explanation in Component 6, but the intuition is:

    Imagine dropping paint at a starting node and watching it spread through the edges. Nearby, well-connected nodes get a lot of paint (high relevance score). Distant, weakly-connected nodes get little paint (low relevance score). PPR computes exactly this — a relevance score for every node in the graph relative to your starting point.

    4. System Overview: The Full Pipeline

    The system has four stages. Each stage takes input from the previous one:

    STAGE 1: PARSE
  Input:  A Python repository (source code files)
  Action: Parse each file into a syntax tree using tree-sitter
  Output: A syntax tree per file (structured representation of the code)

    │
    ▼

STAGE 2: EXTRACT & RESOLVE
  Input:  Syntax trees from Stage 1
  Action: Extract entities (functions, classes, imports) from each tree,
          then resolve cross-file connections (which import points to which file,
          which function call refers to which function definition)
  Output: A list of nodes (entities) and edges (relationships)

    │
    ▼

STAGE 3: BUILD GRAPH & RUN PPR
  Input:  Nodes and edges from Stage 2
  Action: Load into Neo4j graph database, create in-memory projection,
          run Personalized PageRank from seed nodes
  Output: A ranked list of code entities with relevance scores

    │
    ▼

STAGE 4: SERVE
  Input:  The graph database (persistent) + the PPR algorithm (on-demand)
  Action: Expose as an MCP server that AI agents can call
  Output: When an agent asks "what code is relevant to this task?",
          return the top-k most relevant code snippets with their source

    Each stage is explained in detail in the following sections.

    5. Component 1: Parsing Code with tree-sitter

    What tree-sitter is

    tree-sitter is a parser library. A parser reads source code (which is just text) and converts it into a structured tree that represents what the code means syntactically. This structured tree is called an Abstract Syntax Tree (AST).

    Analogy: Parsing is like diagramming a sentence in English class. The sentence "The dog chased the cat" is just a string of characters. A diagram shows its structure: subject ("The dog"), verb ("chased"), object ("the cat"). Similarly, an AST shows code structure: this is a function definition, that's a function call, this is an if-statement.

    What tree-sitter produces

    Given this Python code:

    def authenticate(username, password):
    user = find_user(username)
    if user and check_password(user, password):
        return create_token(user)
    return None

    tree-sitter produces a tree like:

    function_definition
  name: "authenticate"
  parameters: ["username", "password"]
  body:
    assignment
      left: "user"
      right: call(name: "find_user", args: ["username"])
    if_statement
      condition: ...
      body:
        return_statement
          value: call(name: "create_token", args: ["user"])
    return_statement
      value: None

    This tree tells us: there's a function called authenticate, it calls find_user, check_password, and create_token.

    Why tree-sitter and not something else

    • Fast: Parses ~5-10ms per 1,000 lines of code. A 100,000-line codebase takes under 2 seconds.
    • Supports 100+ languages: Python, JavaScript, TypeScript, Java, Go, Rust, and more.
    • Industry standard: Used by GitHub, Cursor, and many other tools.
    • Handles broken code: Doesn't crash on syntax errors — marks the broken parts and keeps going.

    What tree-sitter cannot do

    This is critical to understand:

    • Cannot resolve imports: Sees from utils import helper as text, but doesn't know which file utils refers to.
    • Cannot resolve types: Sees obj.method() but doesn't know which class obj is an instance of.
    • Cannot look across files: Only parses one file at a time. Connecting information across files is our responsibility.

    Tree-sitter gives us the raw material (what entities exist in each file). The next components connect them across files.

    How tree-sitter is used in this project

    For each .py file in the repository:

    import tree_sitter_python as tspython
from tree_sitter import Language, Parser

# Create a Python parser
parser = Parser(Language(tspython.language()))

# Read the source code
with open(file_path, 'rb') as f:
    source = f.read()

# Parse into a syntax tree
tree = parser.parse(source)

# Now tree.root_node gives access to the entire AST

    tree-sitter also provides a query language that lets you search for specific patterns in the tree. For example, to find all function definitions:

    (function_definition name: (identifier) @function.name)

    This says: "find every function_definition node and capture its name child." The @function.name part is like naming a capture group in a regex — it lets you extract the matched piece.

    6. Component 2: Entity Extraction

    What entity extraction is

    After tree-sitter gives us a syntax tree for each file, we walk through it and pull out the entities we care about. An entity is any code element that will become a node in our graph.

    What we extract

    For each Python file, we extract:

    Functions (standalone, not inside a class): 

    def validate_email(email):  # ← this becomes a Function node
    ...
    Captured properties: name, file_path, line_number, end_line, signature (the parameter list), docstring (if present)

    Classes: 

    class AuthService(BaseService):  # ← this becomes a Class node
    ...                           #    with an INHERITS_FROM edge to BaseService
    Captured properties: name, file_path, line_number, end_line, bases (parent classes)

    Methods (functions inside a class): 

    class AuthService:
    def authenticate(self, username, password):  # ← this becomes a Method node
        ...
    Captured properties: same as Functions, plus class_name (which class it belongs to)

    Import statements: 

    from src.utils.helpers import validate_email  # ← records: this file imports validate_email from src.utils.helpers
import json                                    # ← records: this file imports json (standard library, will be skipped)
    Captured: module_path ("src.utils.helpers"), imported_names (["validate_email"]), is_relative (True/False)

    Function calls (inside function/method bodies): 

    def authenticate(self, username, password):
    user = find_user(username)          # ← records: authenticate calls find_user
    if check_password(user, password):  # ← records: authenticate calls check_password
        return create_token(user)       # ← records: authenticate calls create_token
    Captured: caller_name, callee_name, line_number

    What we skip (and why)

    • Standard library imports (import json, import os): These are part of Python, not the project. Including them would add noise without value — every file imports os, and that tells us nothing about the project's structure.
    • Third-party library calls (requests.get(), pandas.DataFrame()): Same reason. We only care about connections within the project's own code.
    • Variables and constants: Not included as nodes for simplicity. Functions and classes capture the meaningful structural relationships. Variables could be added later but would dramatically increase graph size with questionable benefit.
    • Decorators, type annotations, comments: Not extracted as separate entities, though decorators on functions/classes are noted as properties.

    How entity extraction works in practice

    This is a two-step process:

    Step 1: Walk the syntax tree and collect raw entities from each file.

    For each file, the extractor produces a data structure like:

    # Result of extracting from auth_service.py
{
    "file": "src/services/auth_service.py",
    "classes": [
        {"name": "AuthService", "line": 10, "end_line": 85, "bases": ["BaseService"]},
    ],
    "functions": [],  # no standalone functions in this file
    "methods": [
        {"name": "authenticate", "class": "AuthService", "line": 12, "end_line": 30,
         "signature": "(self, username: str, password: str) -> bool"},
        {"name": "refresh_token", "class": "AuthService", "line": 32, "end_line": 50,
         "signature": "(self, token: str) -> str"},
    ],
    "imports": [
        {"module": "src.models.user", "names": ["User"], "is_relative": False},
        {"module": "src.utils.crypto", "names": ["hash_password"], "is_relative": False},
    ],
    "calls": [
        {"caller": "AuthService.authenticate", "callee": "hash_password", "line": 22},
        {"caller": "AuthService.authenticate", "callee": "User.find_by_username", "line": 18},
    ],
}

    Step 2: After processing ALL files, resolve cross-file relationships.

    The raw extraction says authenticate calls hash_password, but we need to know: which file is hash_password defined in? This is where import resolution and call resolution come in (next sections).

    Qualified names

    To uniquely identify entities across the entire codebase, we use qualified names — the full path to an entity:

    • A function validate_email in src/utils/helpers.py → src.utils.helpers.validate_email
    • A method authenticate on class AuthService in src/services/auth_service.py → src.services.auth_service.AuthService.authenticate
    • A class User in src/models/user.py → src.models.user.User

    This prevents confusion when two different files have functions with the same name (e.g., multiple files might have a validate() function).

    7. Component 3: Import Resolution

    What the problem is

    When tree-sitter parses from src.utils.helpers import validate_email, it gives us the string "src.utils.helpers" and the name "validate_email". But tree-sitter does not tell us which actual file on disk "src.utils.helpers" refers to. It only looks at one file at a time and doesn't know about the filesystem.

    Import resolution is the process of turning the string "src.utils.helpers" into the actual file path /project/src/utils/helpers.py.

    Without this step, we can't create IMPORTS edges in our graph because we don't know which files are connected.

    How Python imports work (simplified)

    When Python sees from src.utils.helpers import validate_email, it:

    1. Takes the dotted path src.utils.helpers
    2. Converts dots to directory separators: src/utils/helpers
    3. Looks for either:
    4. A file: src/utils/helpers.py
    5. A package: src/utils/helpers/__init__.py
    6. Inside that file, looks for something named validate_email

    Relative imports start with dots: from ..models import User. Each dot means "go up one directory." So in src/services/auth_service.py, ..models means "go up from services, then into models" → src/models.

    The categories of difficulty

    Easy cases (~80% of real imports in well-structured projects):

    from src.utils.helpers import validate_email
# Resolution: find src/utils/helpers.py in the project → done

    This is just converting dots to slashes and checking if the file exists. Straightforward string manipulation + filesystem check.

    Medium cases (~15%):

    from ..models import User
# Resolution: need to know the current file's location,
# count the dots, go up directories, find models.py or models/__init__.py

    Relative imports require knowing the current file's package context. If you're in src/services/auth.py, .. means src/, so ..models = src/models.

    Another medium case is re-exports through __init__.py: 

    # src/models/__init__.py contains:
from .user import User
from .post import Post

# So "from src.models import User" actually resolves to src/models/user.py

    To fully resolve this, you'd need to read __init__.py and follow the chain. For the initial implementation, resolving to the __init__.py file itself is acceptable — the connection between src/models/ and the importing file is still captured.

    Impossible cases (~5%, skip these):

    module = importlib.import_module(config.module_name)   # runtime string — impossible to resolve statically
exec("from " + module_name + " import something")      # generated code — impossible

    These depend on values only known at runtime. No static analysis tool can resolve them. We log them as unresolved and move on.

    The implementation approach

    Build a custom resolver (~150-200 lines of Python) that handles the easy and medium cases:

    def resolve_import(import_path: str, current_file: str, project_root: str) -> str | None:
    """
    Turn a dotted import path into an actual file path.

    'src.utils.helpers' → '/project/src/utils/helpers.py'
    Returns None if the import can't be resolved (logged as unresolved).
    """
    if import_path.startswith('.'):
        # Relative import: count leading dots, resolve relative to current file
        dots = len(import_path) - len(import_path.lstrip('.'))
        remaining = import_path.lstrip('.')
        # Go up 'dots' directories from current file's directory
        base_dir = current_file_directory
        for _ in range(dots):
            base_dir = parent(base_dir)
        relative_path = remaining.replace('.', '/')
    else:
        # Absolute import: resolve from project root
        relative_path = import_path.replace('.', '/')
        base_dir = project_root

    # Try as a module file (src/utils/helpers.py)
    candidate = base_dir / (relative_path + '.py')
    if exists(candidate):
        return candidate

    # Try as a package (src/utils/helpers/__init__.py)
    candidate = base_dir / relative_path / '__init__.py'
    if exists(candidate):
        return candidate

    return None  # Unresolved — log it and continue

    Why build a custom resolver instead of using an existing tool like jedi:

    The custom resolver is educational (good for thesis) and sufficient for the well-structured repos in SWE-bench (Flask, Django, scikit-learn). If the custom resolver proves too limited (resolving less than ~75% of imports), jedi can be added as a fallback. jedi is a Python library (pip install jedi) that can resolve imports with high accuracy because it understands Python's full import system.

    What we report: The import resolution rate (e.g., "resolved 87% of 610 imports in Flask"). This is a useful metric to include in the thesis — it shows honesty about limitations.

    8. Component 4: Call Resolution

    What the problem is

    Entity extraction tells us: "function authenticate calls something named hash_password". But which hash_password? There might be multiple functions with that name across the codebase. We need to figure out which specific function definition is being called.

    How call resolution works

    Call resolution connects call sites (places where a function is called) to function definitions (where the function is defined). This uses the information from import resolution:

    # In auth_service.py:
from src.utils.crypto import hash_password    # ← import resolution told us this points to src/utils/crypto.py

def authenticate(self, username, password):
    hashed = hash_password(password)           # ← call resolution: "hash_password" was imported from
                                               #   src/utils/crypto.py, so this call resolves to
                                               #   src.utils.crypto.hash_password

    The resolution logic:

    1. When we see a call to some_function() inside a file, check:
    2. Was some_function imported from another file? → Resolve to that file's definition.
    3. Is some_function defined in the same file? → Resolve to the local definition.
    4. Is some_function a method call on self? (like self.validate()) → Resolve to the method on the same class.

    5. Can't determine? → Log as unresolved.

    6. For method calls on objects (like user.save()), the challenge is knowing what type user is. Without type inference, we can use heuristics:

    7. If user was assigned from User.find_by_id(...), we can guess it's a User instance.
    8. If type hints are present (user: User = ...), we can use those.
    9. If neither, log as unresolved.

    What gets created from call resolution

    Each resolved call becomes a CALLS edge in the graph:

    authenticate() ──CALLS──▶ hash_password()
authenticate() ──CALLS──▶ User.find_by_username()
process_login() ──CALLS──▶ authenticate()

    Unresolved calls are counted and reported. For Python projects with good structure and type hints, expect 60-80% resolution. This is acceptable — even partial call graphs are valuable for PPR because the algorithm works with incomplete graphs (it just won't find connections through the missing edges).

    9. Component 5: Building the Graph in Neo4j

    Why Neo4j

    Neo4j is a graph database — a database specifically designed to store and query graph-structured data (nodes and relationships). It's the natural fit for this project because:

    • Our data is a graph (code entities with relationships).
    • We need graph algorithms (PPR) — Neo4j has a built-in Graph Data Science (GDS) library with PPR already implemented.
    • It supports fast graph queries via its query language (Cypher).
    • Free: Neo4j Desktop provides a free Enterprise license for development.
    • The CodexGraph paper (NAACL 2025) used Neo4j for the same kind of code graph, validating this choice.

    The graph schema

    Nodes (4 types):

    Node type What it represents Example Key properties
    File A .py source file auth_service.py file_path, name, line_count
    Class A class definition AuthService name, file_path, line_number, end_line
    Function A standalone function (not in a class) validate_email() name, file_path, line_number, end_line, signature
    Method A function inside a class AuthService.authenticate() name, file_path, line_number, signature, class_name

    Edges (4 types):

    Edge type Connects Meaning Example
    CONTAINS File → Class, Class → Method "This file defines this class" or "This class has this method" auth_service.py CONTAINS AuthService
    IMPORTS File → File "This file imports from that file" auth_service.py IMPORTS user_model.py
    CALLS Function/Method → Function/Method "This function calls that function" authenticate() CALLS hash_password()
    INHERITS_FROM Class → Class "This class extends that class" AuthService INHERITS_FROM BaseService

    Why these 4 edge types and not more:

    This schema follows LocAgent (ACL 2025), which uses 4 entity types and 4 relation types and achieved the best results among the papers we reviewed. The key insight from that paper: you need enough relationship types to capture the important structural connections, but adding too many types adds noise without improving results. CONTAINS, IMPORTS, CALLS, and INHERITS_FROM capture the vast majority of meaningful structural relationships in a Python codebase.

    Loading data into Neo4j

    After entity extraction and resolution, we have lists of nodes and edges. These get loaded into Neo4j using batched operations (loading many records at once, not one at a time):

    # Loading function nodes in a batch (fast — thousands per second)
tx.run("""
    UNWIND $nodes AS node
    MERGE (n:Function {qualified_name: node.qualified_name})
    SET n.name = node.name,
        n.file_path = node.file_path,
        n.line_number = node.line_number,
        n.end_line = node.end_line,
        n.signature = node.signature
""", nodes=function_list)

    MERGE means "create this node if it doesn't exist, or match it if it does." This makes the operation safe to run multiple times (idempotent).

    Performance: Loading 10,000-50,000 nodes with batched operations takes 2-15 seconds. For the thesis-sized repositories (Flask: ~5,000 LOC, Django: ~300,000 LOC), this is fast enough.

    The GDS graph projection

    Before running algorithms, Neo4j's Graph Data Science (GDS) library needs an in-memory projection — a copy of the graph loaded into RAM for fast computation. This is created with:

    CALL gds.graph.project(
  'codeGraph',                                    -- name for this projection
  ['Function', 'Method', 'Class', 'File'],        -- which node types to include
  {
    CALLS:         {orientation: 'UNDIRECTED', properties: 'weight'},
    IMPORTS:       {orientation: 'UNDIRECTED', properties: 'weight'},
    INHERITS_FROM: {orientation: 'UNDIRECTED', properties: 'weight'},
    CONTAINS:      {orientation: 'UNDIRECTED', properties: 'weight'}
  }
)

    Why UNDIRECTED: Explained in Component 6 — Directionality.

    Memory usage: A 50,000-node graph with 200,000 edges needs approximately 5.2 MB in-memory. Trivial by modern laptop standards.

    10. Component 6: Personalized PageRank (PPR)

    This is the core algorithm of the project. This section explains it from scratch — no prior knowledge assumed.

    Starting with regular PageRank

    PageRank was invented by Larry Page and Sergey Brin (the founders of Google) in 1998 to rank web pages. The idea:

    Imagine a person randomly browsing the web. They start on a random page, then randomly click one of the links on that page, then randomly click a link on the next page, and so on. The pages they visit most often are the most "important" ones.

    Why? Because important pages have many incoming links from other important pages. If lots of pages link to Page X, the random surfer will end up at Page X frequently.

    There's one catch: what if the surfer reaches a page with no outgoing links, or gets stuck in a loop? To handle this, with some probability (typically 15%), the surfer "teleports" — jumps to a completely random page instead of following a link. This ensures they eventually visit every page.

    The result of PageRank is a score for every page representing how often the random surfer visits it. Higher score = more important.

    What makes Personalized PageRank different

    In regular PageRank, when the surfer teleports, they jump to any random page with equal probability. This produces a global importance ranking.

    In Personalized PageRank (PPR), when the surfer teleports, they jump back to specific pages you choose — called seed nodes. This biases the walk toward a particular region of the graph.

    Analogy: Regular PageRank asks "what are the most important pages on the entire web?" PPR asks "what are the most important pages relative to this specific page?"

    For code: Regular PageRank would tell you the most important functions in the entire codebase (useful but not task-specific). PPR from authenticate() tells you the most important functions relative to authenticate() — the ones most structurally connected to it.

    How PPR works step by step

    Given a graph and a seed node (e.g., authenticate()):

    1. Start at the seed. Place all "probability mass" (think of it as 1.0 units of paint) on the seed node.

    2. Walk step: From each node that has paint, spread the paint along outgoing edges. If a node has 3 outgoing edges, each edge gets 1/3 of the paint. But only spread (1 - α) fraction of the paint — keep α fraction at the seed.

    3. α (alpha) is called the "restart probability" or "teleportation probability." Typically 0.15 (15%). It controls how far the paint spreads from the seed. Higher α = paint stays closer to seed. Lower α = paint spreads further.

    4. Repeat steps 1-2 many times (typically 20 iterations). The paint distribution converges — it stops changing significantly.

    5. Read the scores. The amount of paint at each node is its PPR score = its relevance to the seed.

    Concrete example:

    Seed: authenticate()

After PPR:
  authenticate()      → 0.150  (high — it's the seed itself)
  hash_password()     → 0.082  (high — directly called by seed)
  check_token()       → 0.067  (high — directly called by seed)
  User.find_by_name() → 0.045  (medium — called by seed)
  BaseService.init()  → 0.023  (medium — inherited by seed's class)
  process_login()     → 0.018  (medium — calls the seed)
  log_error()         → 0.012  (lower — but still connected, see "utility function problem" below)
  send_email()        → 0.001  (very low — barely connected)

    The top-ranked results are the code entities most structurally relevant to authenticate().

    The key parameters

    Parameter What it controls Default Effect of increasing
    α (alpha) or restart probability How close results stay to the seed 0.15 Higher → results cluster more tightly around seed, fewer distant results
    maxIterations How many times the algorithm repeats 20 Higher → more precise scores (diminishing returns after ~20)
    tolerance When to stop early if scores stop changing 1e-7 Lower → more precise but slower (irrelevant at thesis scale)
    top-k How many results to return 10-20 Higher → more context but more noise and more tokens used

    In Neo4j GDS, the dampingFactor parameter is (1 - α). So dampingFactor = 0.85 means α = 0.15 — a 15% chance of teleporting back to the seed at each step.

    Why PPR is the right algorithm for this problem

    Compared to simpler approaches:

    Approach How it works Limitation
    1-hop neighbors Return everything directly connected to the seed Misses indirect but important connections (A→B→C — C is missed)
    k-hop ego-graph (what RepoGraph uses) Return everything within k edges of the seed k=1 is too narrow; k=2 explodes in size (too many results); no ranking within results
    Shortest path Find entities closest to seed by edge count Doesn't account for multiple paths or connection strength
    PPR (this project) Probabilistic relevance that considers all paths, multiple connections, and connection strength Slightly more complex to implement — but Neo4j GDS handles it

    PPR's advantages: - Multi-hop: Finds indirectly connected but important code. - Ranked: Every node gets a score, so you can take exactly the top-k you need. - Multiple paths matter: If node X is connected to the seed through 5 different paths, it scores higher than node Y connected through only 1 path. This reflects real code coupling. - Mathematically principled: Guaranteed to converge to a unique solution. Not a heuristic — it's provably correct. - Fast: 1-10 milliseconds on 50,000-node graphs. Faster than the network roundtrip to the database.

    Directionality: which direction should PPR walk?

    Edges in the code graph have a direction. authenticate() CALLS hash_password() is an edge from authenticate to hash_password.

    When PPR walks from the seed, which direction does it follow?

    Direction What it means What PPR finds Example
    Natural (follow edge direction) Walk from caller to callee Things the seed depends on (downstream) "authenticate calls hash_password" → finds hash_password
    Reverse (follow edges backward) Walk from callee to caller Things that depend on the seed (upstream) "process_login calls authenticate" → finds process_login
    Undirected (follow both ways) Walk in both directions Everything structurally connected Finds both hash_password AND process_login

    Decision: UNDIRECTED as the default. Reason: for context selection, you want to see both what a function uses (so you understand it) and what uses it (so you know what might break). An AI agent fixing authenticate() needs to see hash_password() (to understand the implementation) AND process_login() (to check if the fix affects callers).

    This is configured in the GDS graph projection by setting orientation: 'UNDIRECTED' — Neo4j treats every edge as bidirectional.

    In the evaluation phase, we will also test directed and reversed PPR as ablation experiments to measure which direction works best. This comparison is interesting thesis material.

    Running PPR in Neo4j

    -- Find the seed node
MATCH (seed:Function {name: 'authenticate', file_path: 'src/services/auth_service.py'})

-- Run PPR from that seed
CALL gds.pageRank.stream('codeGraph', {
  maxIterations: 20,
  dampingFactor: 0.85,
  sourceNodes: [seed],
  relationshipWeightProperty: 'weight'
})
YIELD nodeId, score

-- Return top 20 results
RETURN gds.util.asNode(nodeId).name AS name,
       gds.util.asNode(nodeId).file_path AS file,
       score
ORDER BY score DESC
LIMIT 20

    PPR can also accept multiple seed nodes with different weights:

    MATCH (a:Function {name: 'authenticate'}), (b:Function {name: 'refresh_token'})
CALL gds.pageRank.stream('codeGraph', {
  maxIterations: 20,
  dampingFactor: 0.85,
  sourceNodes: [a, b]
})

    This distributes paint equally across both seeds. GDS also supports weighted seeds if you want to give more importance to one seed over another.

    11. Component 7: Seed Node Selection

    The core question

    PPR starts from seed nodes and spreads outward. The quality of your seeds directly determines the quality of your results. Good seeds → relevant context. Bad seeds → useless context.

    The seed selection problem: how does the system decide where to start PPR?

    Why "current file" is a bad default seed

    A naïve approach would be: "the developer is editing auth_service.py, so use all functions in that file as seeds." This fails because:

    • The developer might have been searching for an hour and landed on a random file that isn't central to their task.
    • The developer might be reading code to understand it, not editing the part they need to change.
    • The AI agent might be exploring multiple files as part of a multi-step task.

    "Where the cursor is" does not reliably indicate "what the developer cares about."

    The actual approach: the AI agent provides the task context

    The MCP server doesn't guess the seed — the AI agent calling the tool tells it what the task is. When a developer says "fix the authentication timeout bug" to Claude Code, the agent already understands the task. It passes that understanding to our tool:

    @mcp.tool()
def get_relevant_context(
    task_description: str,                  # "fix the authentication timeout bug"
    mentioned_entities: list[str] = None,   # ["authenticate", "TokenExpiredError"]
    current_file: str = None                # optional low-weight hint
) -> str:

    The tool then extracts seeds from this information.

    The seed extraction pipeline

    Seeds are selected through a multi-step process, combining multiple signals with different confidence levels:

    Step 1: Extract explicitly mentioned entities
  Input:  task_description = "fix the authentication timeout in AuthService"
          mentioned_entities = ["authenticate", "AuthService", "TokenExpiredError"]
  Action: Match these names against nodes in the graph
  Result: AuthService node (exact match), authenticate method (exact match)
  Weight: HIGH (these were explicitly named)

Step 2: BM25 keyword search as supplement
  Input:  task_description text searched against all function signatures and docstrings
  Action: Standard keyword matching (BM25) ranks functions by textual relevance to the task
  Result: Top 3-5 functions that mention "timeout", "token", "auth", etc.
  Weight: MEDIUM (text relevance, not as reliable as explicit mentions)

Step 3: Current file as weak hint (if provided)
  Input:  current_file = "src/services/auth_service.py"
  Action: Add functions from this file as seeds
  Weight: LOW (might be relevant, might be where the developer happened to be browsing)

Step 4: Build the personalization vector
  Action: Combine all seeds with their weights, normalize so they sum to 1.0
  Result: {AuthService.authenticate: 0.35, AuthService: 0.25,
           timeout_handler: 0.15, token_validator: 0.10,
           AuthService.refresh_token: 0.08, AuthService.logout: 0.07}
  This becomes the personalization vector for PPR.

    BM25 explained: BM25 is a text search algorithm. Given a query ("authentication timeout") and a set of documents (all function signatures and docstrings in the codebase), it ranks documents by how well they match the query terms. It considers: does the document contain the query words? How many times? How long is the document compared to average? BM25 is the standard baseline in information retrieval — simple, fast, and surprisingly effective.

    Why this multi-signal approach matters

    Different scenarios benefit from different signals:

    Scenario Best signal Why
    Developer says "fix bug in authenticate()" Explicit entity extraction Clear function name mentioned
    Developer says "the login flow is broken" BM25 keyword search No specific function named, but keywords help
    Developer says "continue working on the auth module" Current file + BM25 Vague task, file hint helps narrow scope
    SWE-bench evaluation (bug report text) Entity extraction + BM25 Bug reports often mention specific classes/functions

    By combining all signals with weights, the system degrades gracefully — even when one signal is weak, the others compensate.

    12. Component 8: Post-Processing the Results

    PPR returns a ranked list of nodes with scores. Before sending these to the AI agent, two post-processing steps improve quality.

    Problem 1: The Utility Function Problem

    What happens: A function like log_error() is called by 200 other functions across the codebase. Because it's connected to everything, PPR from any seed will rank it highly. But log_error() is not useful context for understanding a specific bug — it's generic infrastructure code.

    Analogy: In Google Search, if "Wikipedia" appeared as the #1 result for every query, that would be technically correct (Wikipedia is linked from everywhere) but not helpful. In text search, this is solved by downweighting common words ("the", "is", "and"). The same idea applies to graphs.

    Solution: IDF-inspired edge weight adjustment.

    IDF stands for Inverse Document Frequency — a concept from text search. The idea: terms that appear in many documents are less informative than terms that appear in few documents. "The" appears everywhere → low IDF → low weight. "Authenticate" appears in few files → high IDF → high weight.

    Applied to our graph: functions called by many other functions are less informative than functions called by few. So we adjust edge weights:

    adjusted_weight(A → B) = base_weight(edge_type) / log2(in_degree(B) + 2)

    Where in_degree(B) is the number of incoming edges to node B (how many functions call B).

    Function in_degree Weight adjustment Effect
    log_error() 200 base / log2(202) ≈ base / 7.7 Heavily downweighted
    hash_password() 3 base / log2(5) ≈ base / 2.3 Slightly downweighted
    calculate_token_expiry() 1 base / log2(3) ≈ base / 1.6 Barely affected

    This means PPR will naturally reduce the score of utility functions while preserving the scores of specific, task-relevant functions.

    Problem 2: Token budgeting

    PPR returns nodes (function/class names with scores). The AI agent needs to see actual source code, not just names. But we can't send unlimited code — there's a token budget.

    The process:

    1. Take the top-k results from PPR (e.g., top 20)
    2. For each result, read the actual source code from the file (using the stored file_path, line_number, end_line)
    3. Accumulate results until reaching the token budget (default: 4,000-8,000 tokens)
    4. Format and return
    def format_context(ppr_results, token_budget=6000):
    context_parts = []
    tokens_used = 0

    for node, score in ppr_results:
        # Read the actual source code for this entity
        source_code = read_lines(node.file_path, node.line_number, node.end_line)
        source_tokens = count_tokens(source_code)

        if tokens_used + source_tokens > token_budget:
            break  # Stop — we've filled the budget

        context_parts.append({
            "entity": node.name,
            "file": node.file_path,
            "lines": f"{node.line_number}-{node.end_line}",
            "relevance_score": round(score, 4),
            "source_code": source_code
        })
        tokens_used += source_tokens

    return context_parts

    Why 4,000-8,000 tokens: The AI agent has its own context window to manage (100K-200K tokens). Our context is one of many inputs. Providing 4,000-8,000 tokens of highly relevant code is more useful than 50,000 tokens of loosely relevant code — the agent can focus on what matters.

    13. Component 9: The MCP Server

    What MCP is

    MCP (Model Context Protocol) is an open standard introduced by Anthropic in November 2024 that defines how AI applications communicate with external tools and data sources.

    Analogy: Before USB, every device had its own unique connector. USB standardized this — any USB device plugs into any USB port. MCP does the same for AI tools: any MCP-compliant tool plugs into any MCP-aware AI application.

    Without MCP, you'd need to build a separate integration for Claude Desktop, another for Cursor, another for Copilot. With MCP, you build one server and it works with all of them.

    How MCP works

    MCP follows a client-server model:

    • The server (our code) provides tools — functions the AI can call.
    • The host (Claude Desktop, Cursor, etc.) connects to the server and lets the AI use those tools.
    • Communication happens via JSON-RPC (a simple request/response protocol) over either STDIO (for local tools) or HTTP (for remote tools).

    Each tool has: - A name: What the tool is called (e.g., get_relevant_context) - A description: What it does (the AI reads this to decide when to use it) - A parameter schema: What inputs it accepts (defined as JSON Schema) - A return value: What it sends back

    The tools this MCP server exposes

    Tool 1: get_relevant_context (the primary tool)

    @mcp.tool()
def get_relevant_context(
    task_description: str,
    mentioned_entities: list[str] = None,
    current_file: str = None,
    top_k: int = 15,
    token_budget: int = 6000
) -> str:
    """
    Find the most structurally relevant code for a given task.

    Uses Personalized PageRank on the code dependency graph to find
    functions, classes, and methods that are structurally connected
    to the entities involved in the task.

    Args:
        task_description: What the developer is trying to do
        mentioned_entities: Specific function/class names from the conversation
        current_file: The file currently open (optional, low-weight hint)
        top_k: Maximum number of results to return
        token_budget: Maximum tokens of source code to include

    Returns:
        JSON array of relevant code snippets with file paths, line numbers,
        relevance scores, and source code.
    """
    # 1. Extract seed nodes from the inputs
    seeds = extract_seeds(task_description, mentioned_entities, current_file)

    # 2. Run PPR from the seeds
    ppr_results = run_ppr(seeds, top_k=top_k)

    # 3. Post-process (filter utility functions, read source code, apply token budget)
    context = format_context(ppr_results, token_budget=token_budget)

    return json.dumps(context)

    14. How Evaluation Works

    What we're measuring

    The fundamental question: Does PPR retrieve the right code?

    We answer this by using real bug reports where we know which files were actually changed to fix the bug. If PPR's top results include those files, PPR "got it right."

    The benchmark: SWE-bench Lite

    SWE-bench Lite is a dataset of 300 real bug reports from popular Python repositories (Django, Flask, scikit-learn, matplotlib, etc.). For each bug, the dataset provides:

    • The repository state before the fix (a specific git commit)
    • The bug description (the GitHub issue text)
    • The gold patch — the actual code changes the developer made to fix the bug (which files were modified, which functions were changed)

    We don't fix the bugs. We don't run an LLM. We just ask: given this bug description, does PPR point to the files that were actually changed?

    The evaluation process

    For each of the 300 bug reports:
  1. Check out the repository at the "before fix" commit
  2. Build the code graph (tree-sitter → Neo4j)
  3. Extract seed nodes from the bug description text
  4. Run PPR from the seeds, get top-k results
  5. Check: do the top-k results include the files from the gold patch?
  6. Record hit/miss and rank of first correct result

    The metrics

    Recall@k — "Out of all the files that needed changing, what percentage appeared in my top k results?"

    Example: A bug required changing 3 files. PPR's top 10 results included 2 of them. → Recall@10 = 2/3 = 66.7%

    We report Recall@5 and Recall@10.

    MRR (Mean Reciprocal Rank) — "On average, where did the first correct file appear in my ranking?"

    If the first correct file is ranked #1 → score = 1/1 = 1.0 If the first correct file is ranked #3 → score = 1/3 = 0.33 If the first correct file is ranked #7 → score = 1/7 = 0.14

    MRR is the average of these scores across all 300 bug reports. Higher MRR means the correct files appear earlier in the ranking.

    What we compare against (baselines)

    Baseline How it works Why include it
    Random Pick k random files from the repo Floor — any reasonable method must beat this
    BM25 Keyword search: match bug description text against code file contents Standard text-retrieval baseline that every paper compares against
    1-hop ego-graph Find entities mentioned in the bug, return their direct graph neighbors (no PPR) Tests whether PPR's multi-hop propagation adds value over simple graph lookup

    Ablation studies

    An ablation study is an experiment where you remove or change one component to measure its impact. It answers: "how much does each part of the system contribute?"

    Planned ablations:

    Experiment What changes What it reveals
    Remove CALLS edges Graph has only IMPORTS, INHERITS, CONTAINS How important are call relationships?
    Remove IMPORTS edges Graph has only CALLS, INHERITS, CONTAINS How important are import relationships?
    Remove INHERITS edges Graph has only CALLS, IMPORTS, CONTAINS How important is inheritance?
    Change α (restart probability) α ∈ How far should PPR spread from the seed?
    Change direction Directed vs reversed vs undirected Which edge direction works best for context?
    With vs without IDF weighting Compare utility-function handling on vs off Does downweighting common functions help?
    PPR vs 1-hop Compare PPR against simple neighbor lookup Does the algorithm's sophistication matter?
    Change top-k k ∈ How many results are needed for good recall?

    Each ablation is a separate run of the evaluation with one variable changed. This produces a table of results that shows what matters and what doesn't — valuable thesis content regardless of the outcome.

    16. Known Limitations

    Being upfront about limitations demonstrates maturity and is expected in academic work.

    1. Static analysis only. The graph is built by reading source code, not by running it. This means dynamic behaviors (runtime type changes, monkey-patching, eval()) are invisible. For well-structured Python projects, static analysis captures the majority of relationships. For highly dynamic codebases, the graph will be incomplete.

    2. Python only (in this thesis). The architecture is designed to be language-extensible (tree-sitter supports 100+ languages, and the graph schema is language-agnostic). But the implementation only covers Python. Adding TypeScript, Java, etc. is future work.

    3. Import resolution is imperfect. The custom resolver handles ~80% of imports in well-structured projects. Complex import patterns (__init__.py re-exports, namespace packages, conditional imports) may be missed. This means some IMPORTS edges will be absent from the graph.

    4. Call resolution is imperfect. Without type inference, method calls on objects (obj.method()) can only be resolved when the type of obj is explicitly known (via type hints or direct assignment). Duck-typed calls will be missed.

    5. Utility functions may still be noisy. The IDF-inspired weighting reduces but may not eliminate the problem of highly-connected infrastructure functions appearing in results. The threshold and formula may need tuning per-codebase.

    6. MCP adoption is still growing. As of February 2026, MCP is supported by Claude Desktop, has been adopted by OpenAI, and is supported in Cursor through extensions. But not every AI coding tool supports MCP natively yet. The system also provides a REST API fallback.

    7. No real-time incremental updates. The graph is built once for a repository snapshot. If the code changes, the graph needs to be rebuilt (takes 5-30 seconds for thesis-sized repos). Real-time incremental updates that reflect each keystroke are out of scope.

    8. PPR may not outperform simpler methods. It's possible that BM25 or 1-hop ego-graphs perform comparably to PPR for some tasks. This would be a valid negative result — the evaluation will reveal it, and the ablation studies will show why.

    17. Glossary

    Term Definition
    AST (Abstract Syntax Tree) A tree-shaped representation of source code's structure. Each node represents a code construct (function, class, expression). Created by a parser.
    BM25 A text search algorithm that ranks documents by keyword relevance. Standard baseline in information retrieval research.
    Cypher Neo4j's query language (like SQL for graphs). Used to create, read, update, and delete nodes and relationships.
    Damping factor In PageRank, the probability of following a link (vs. teleporting). Typically 0.85. In PPR terms: damping = 1 - α.
    Edge A connection between two nodes in a graph. In this project: CALLS, IMPORTS, INHERITS_FROM, CONTAINS.
    Embedding A list of numbers (vector) that represents the "meaning" of a piece of text or code. Used by Cursor/Copilot for similarity search.
    Entity A code element that becomes a node in the graph: a function, class, method, or file.
    GDS (Graph Data Science) Neo4j's plugin library that provides graph algorithms (PageRank, PPR, community detection, etc.)
    Gold patch In SWE-bench, the actual code changes a developer made to fix a bug. Used as the "correct answer" for evaluation.
    Graph projection An in-memory copy of part of the Neo4j graph, optimized for running algorithms. Created before running PPR.
    IDF (Inverse Document Frequency) A weighting technique that gives lower weight to terms/nodes that appear frequently (they're less informative).
    Import resolution The process of determining which actual file a Python import statement refers to.
    MCP (Model Context Protocol) An open standard by Anthropic for connecting AI applications to external tools. Like USB-C for AI tools.
    MRR (Mean Reciprocal Rank) An evaluation metric: the average of (1 / rank of first correct result) across all queries.
    Node An entity in a graph. In this project: a File, Class, Function, or Method.
    PPR (Personalized PageRank) A graph algorithm that ranks nodes by relevance to specific seed nodes, using a random walk with restart.
    Qualified name The full, unique identifier for a code entity (e.g., src.services.auth.AuthService.authenticate).
    Recall@k An evaluation metric: what fraction of all correct answers appeared in the top k results.
    Seed node The starting point for PPR. The node(s) you're computing relevance relative to.
    SWE-bench A benchmark of real GitHub bug reports with known fixes, used to evaluate code retrieval and fixing tools.
    Token A unit of text for LLMs. Roughly 3/4 of a word. A "token budget" limits how much text you send.
    tree-sitter An open-source parser library that converts source code into syntax trees. Fast, supports 100+ languages.

    Edge Weight Configuration Reference

    The complete edge weight system combines a base weight per relationship type with an IDF adjustment based on the target node's connectivity:

    final_weight(A → B) = base_weight(edge_type) / log2(in_degree(B) + 2)

    Base weights by relationship type:

    Edge type Base weight Rationale
    INHERITS_FROM 1.0 Strongest coupling — changing a base class affects all subclasses
    CALLS 0.8 Direct dependency — if the callee changes, the caller may break
    IMPORTS 0.5 File-level connection — weaker than function-level call
    CONTAINS 0.4 Structural containment — relevant but not a dependency

    IDF adjustment examples:

    Target node in_degree log2(in_degree + 2) Multiplier (1/log)
    calculate_expiry() (called by 1 function) 1 1.58 0.63 (barely affected)
    validate_input() (called by 8 functions) 8 3.32 0.30 (moderately reduced)
    log_error() (called by 200 functions) 200 7.66 0.13 (heavily reduced)

    These weights are applied when creating the GDS graph projection and are used by PPR during the random walk.

    Performance Expectations

    Operation Expected time (10k-50k node graph)
    tree-sitter parsing (100k LOC) 0.5-2 seconds
    Entity extraction + resolution 1-5 seconds
    Neo4j graph loading (batched) 2-15 seconds
    GDS projection creation < 1 second
    PPR execution 1-10 milliseconds
    Source code reading + formatting 50-200 milliseconds
    Total: full pipeline (parse → result) 5-30 seconds
    Total: query only (graph already built) < 250 milliseconds

    The graph only needs to be built once per repository state. Subsequent queries (PPR + formatting) take under 250ms — well within interactive use.
