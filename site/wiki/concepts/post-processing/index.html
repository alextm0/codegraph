    Post-Processing

    PPR returns a ranked list of nodes with scores. Before these reach the AI agent, two post-processing steps improve quality: IDF-based edge reweighting and token budgeting.

    Problem 1: The Utility Function Problem

    What Happens Without This Step

    A function like log_error() is called by 200 other functions across the codebase. Because it has 200 incoming edges, PPR from any seed node will rank it highly — it's structurally well-connected. But log_error() is not useful context. It's generic infrastructure.

    Analogy: If "Wikipedia" appeared as the #1 search result for every Google query, that's technically correct (Wikipedia is linked from everywhere) but useless.

    Solution: IDF-Inspired Edge Weight Adjustment

    IDF (Inverse Document Frequency) is a concept from text search: terms appearing in many documents are less informative. "The" appears everywhere → low weight. "Authenticate" appears rarely → high weight.

    We apply the same logic to graph edges: edges to highly-connected nodes are downweighted.

    adjusted_weight(A → B) = base_weight(edge_type) / log₂(in_degree(B) + 2)

    Where in_degree(B) = number of incoming edges to node B.

    The Effect

    Function In-degree Weight factor Outcome
    log_error() 200 base / log₂(202) ≈ base / 7.7 Heavily downweighted
    hash_password() 3 base / log₂(5) ≈ base / 2.3 Slightly downweighted
    calculate_token_expiry() 1 base / log₂(3) ≈ base / 1.6 Barely affected

    Generic utility functions see much less weight flowing into them via PPR, so they score lower and are less likely to appear in the final top-k.

    When It's Applied

    IDF reweighting is applied to edge weights before the GDS projection is created, not after PPR runs. This means PPR already operates on the adjusted weights — the downweighting is baked into the walk itself.

    # src/retrieval/post_processing.py
def apply_idf_weights(
    edges: list[Edge],
    in_degree: dict[str, int]
) -> list[Edge]:
    """Adjust edge weights by in-degree of target node."""
    for edge in edges:
        target_degree = in_degree.get(edge.target, 0)
        edge.weight = edge.weight / math.log2(target_degree + 2)
    return edges

    Problem 2: Token Budgeting

    The Challenge

    PPR returns node names with scores. The AI agent needs actual source code, not just names. But we can't send unlimited code — context windows have limits, and too much irrelevant code buries the signal.

    The Process

    1. Take the top-k results from PPR (default: 20 nodes)
    2. For each node, read source code using the stored file_path, line_number, end_line
    3. Count tokens in the source code
    4. Add nodes to the result until the token budget is consumed
    5. Format and return
    def format_context(
    ppr_results: list[PPRResult],
    token_budget: int = 6000
) -> list[ContextResult]:
    context_parts = []
    tokens_used = 0

    for node, score in ppr_results:
        source_code = read_lines(node.file_path, node.line_number, node.end_line)
        source_tokens = count_tokens(source_code)

        if tokens_used + source_tokens > token_budget:
            break   # Budget consumed — stop adding

        context_parts.append(ContextResult(
            entity_name=node.name,
            qualified_name=node.qualified_name,
            file_path=node.file_path,
            line_number=node.line_number,
            source_code=source_code,
            relevance_score=round(score, 4)
        ))
        tokens_used += source_tokens

    return context_parts

    Why 4,000–8,000 Tokens?

    The AI agent (Claude, GPT-4, etc.) manages its own context window (100K–200K tokens). CodeGraph's contribution is one part of that context. 4,000–8,000 tokens of highly relevant code is more useful than 50,000 tokens of loosely relevant code — the agent can focus on what matters.

    The default token budget is 6,000, configurable in config.yaml.

    Output: ContextResult

    Each item in the final output:

    @dataclass
class ContextResult:
    entity_name: str        # e.g., "authenticate"
    qualified_name: str     # e.g., "src/services/auth_service.py::AuthService.authenticate"
    file_path: str          # e.g., "src/services/auth_service.py"
    line_number: int        # start line in the file
    end_line: int           # end line in the file
    source_code: str        # the actual Python source
    relevance_score: float  # PPR score (higher = more relevant)

    Configuration

    # config.yaml
mcp:
  default_token_budget: 6000   # Max tokens in the MCP response
  default_top_k: 15            # How many PPR results to consider before budgeting
