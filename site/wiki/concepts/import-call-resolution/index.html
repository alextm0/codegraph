    Import & Call Resolution

    tree-sitter gives us entities within each file. Resolution connects them across files — turning raw strings like "src.utils.helpers" into actual edges in the graph.

    This is the trickiest part of the pipeline and the primary source of incomplete graphs.

    The Two Resolution Problems

    Problem 1: Import Resolution

    tree-sitter parses from src.utils.crypto import hash_password and gives us the string "src.utils.crypto". It does not tell us which file on disk this refers to.

    Import resolution turns "src.utils.crypto" → src/utils/crypto.py

    Without this, we can't create IMPORTS edges because we don't know which files are connected.

    Problem 2: Call Resolution

    Entity extraction tells us: "authenticate() calls something named hash_password." But which hash_password? There may be multiple functions with that name across the codebase.

    Call resolution connects a call site (e.g., hash_password(password) in auth_service.py) to the actual function definition (e.g., hash_password in src/utils/crypto.py).

    Import Resolution: How It Works

    Python's Import System (Simplified)

    When Python sees from src.utils.crypto import hash_password:

    1. Take src.utils.crypto
    2. Replace dots with slashes: src/utils/crypto
    3. Look for either a module file (src/utils/crypto.py) or a package (src/utils/crypto/__init__.py)
    4. hash_password is then an attribute of that module

    Relative imports start with dots: from ..models import User. Each dot means "go up one directory." In src/services/auth_service.py, ..models resolves to src/models.

    Difficulty Tiers

    Category Frequency Example
    Easy ~80% from src.utils.helpers import validate_email — direct path mapping
    Medium ~15% Relative imports (from ..models import User), re-exports via __init__.py
    Impossible ~5% importlib.import_module(config.module_name) — runtime dynamic imports

    The impossible cases are logged as unresolved and skipped — returning partial results is better than crashing.

    The Resolver Algorithm

    def _resolve_import_to_file_path(module_path: str, current_file: str, project_root: str) -> str | None:
    if module_path.startswith('.'):
        # Relative import — count leading dots, go up directories
        dots = len(module_path) - len(module_path.lstrip('.'))
        remaining = module_path.lstrip('.')
        base_dir = Path(current_file).parent
        for _ in range(dots - 1):
            base_dir = base_dir.parent
        relative_path = remaining.replace('.', '/')
    else:
        # Absolute import — resolve from project root
        relative_path = module_path.replace('.', '/')
        base_dir = Path(project_root)

    # Try as a module file: src/utils/crypto.py
    candidate = base_dir / (relative_path + '.py')
    if candidate.exists():
        return str(candidate)

    # Try as a package: src/utils/crypto/__init__.py
    candidate = base_dir / relative_path / '__init__.py'
    if candidate.exists():
        return str(candidate)

    # Unresolved — log and continue
    return None

    Path boundary check: When matching against known file paths, the resolver checks fp == suffix or fp.endswith("/" + suffix) to avoid false matches on partial path segments.

    Call Resolution: How It Works

    Call resolution uses import resolution results as a key input.

    Resolution Priority

    When a call site callee_name appears inside a function in some_file.py:

    1. Import match — Was callee_name imported from another file? If so, resolve to that file's definition.
    2. Same-file match — Is callee_name defined in the same file?
    3. Unique global match — Does only one entity with that name exist in the entire codebase?
    4. Ambiguous / unresolved — Multiple candidates, none clearly correct. Log and skip.

    Implementation: Entity Lookup

    def _build_entity_lookup(all_entities: list[FileEntities]) -> dict[str, list[str]]:
    """Returns dict[name → list of qualified_names] — all candidates per name."""
    lookup = {}
    for file_entities in all_entities:
        for func in file_entities.functions + file_entities.methods:
            lookup.setdefault(func.name, []).append(func.qualified_name)
    return lookup

def _build_import_map(file_entities: FileEntities, resolved_imports: dict) -> dict[str, str]:
    """Maps imported names → resolved file path for this file."""
    import_map = {}
    for imp in file_entities.imports:
        if imp.module_path in resolved_imports:
            for name in imp.imported_names:
                import_map[name] = resolved_imports[imp.module_path]
    return import_map

    Method Call Heuristics

    For self.method() calls, the class is known from context, so these resolve cleanly. For obj.method() calls on local variables, the resolver uses:

    • Type hints if present (user: User = ...)
    • Assignment inference (user = User.find_by_id(...) → assume User instance)
    • Otherwise: logged as unresolved

    Resolution Rates and Expectations

    For well-structured Python projects (like those in SWE-bench):

    Resolution Type Expected Rate
    Import resolution ~80–90%
    Call resolution ~60–80%

    These rates are logged during graph building and reported as an honesty metric. PPR works with incomplete graphs — missing edges just mean some connections aren't captured, but the overall ranking is still meaningful.

    Why Not Use jedi?

    jedi is a Python type inference library that can resolve imports and types much more accurately (near 100%). We use a custom resolver instead for two reasons:

    1. Educational value — understanding the resolution algorithm is important for the thesis
    2. Sufficient accuracy — SWE-bench repositories are well-structured; ~80% resolution produces good PPR results

    If resolution falls below 70% for a particular repository, jedi can be added as a fallback.
