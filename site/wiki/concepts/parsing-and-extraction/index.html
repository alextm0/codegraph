    Parsing and Entity Extraction

    This page covers how CodeGraph reads Python source files and pulls out the entities that become nodes in the graph.

    The Two-Step Process

    Step 1: Parsing — Convert raw source text into a structured Abstract Syntax Tree (AST) using tree-sitter.

    Step 2: Extraction — Walk the AST and collect functions, classes, methods, imports, and call sites.

    These two steps together produce FileEntities — the structured representation of a single source file, which feeds directly into the graph builder.

    What Is tree-sitter?

    tree-sitter is a parser library that turns source code (text) into an Abstract Syntax Tree (AST) — a structured representation of the code's syntax.

    Think of it like parsing a sentence:

    "The dog chased the cat"
→ subject("The dog") + verb("chased") + object("the cat")

    tree-sitter does the same for code:

    def authenticate(username, password):
    user = find_user(username)
    return create_token(user)

    Becomes:

    function_definition
  name: identifier → "authenticate"
  parameters: ["username", "password"]
  body:
    assignment → user = call(find_user, [username])
    return_statement → call(create_token, [user])

    Why tree-sitter?

    Property Value
    Speed ~5–10ms per 1,000 lines
    Error tolerance Handles broken/incomplete code, keeps parsing
    Multi-language 100+ languages with the same API
    Industry adoption Used by GitHub, Cursor, Neovim
    Query language Pattern matching syntax (like regex for ASTs)

    What tree-sitter Cannot Do

    • Cannot resolve imports — sees from utils import helper as text, not as a file reference
    • Cannot determine types — sees obj.method() but doesn't know which class obj is
    • Cannot look across files — parses one file at a time

    Cross-file resolution is handled separately. See Import & Call Resolution →.

    How tree-sitter Is Used in This Project

    # src/parser/python_parser.py
import tree_sitter_python as tspython
from tree_sitter import Language, Parser

def create_parser() -> Parser:
    return Parser(Language(tspython.language()))

def parse_file(file_path: str, parser: Parser) -> FileEntities:
    with open(file_path, 'rb') as f:
        source = f.read()
    tree = parser.parse(source)
    return extract_entities(tree, file_path, source)

    tree-sitter also provides a pattern-matching query language:

    ; Find every function definition and capture its name
(function_definition name: (identifier) @function.name)

; Find every class definition
(class_definition name: (identifier) @class.name)

; Find function calls
(call function: (identifier) @call.name)

    This is how the extractor identifies nodes of interest without manually walking the AST tree.

    What Gets Extracted

    From each .py file, the extractor (src/parser/extractors.py) pulls out:

    Functions (top-level)

    def validate_email(email: str) -> bool:
    """Check if the email is valid."""
    ...

    Captured: name, file_path, line_number, end_line, signature, docstring

    Classes

    class AuthService(BaseService):
    """Handles user authentication."""
    ...

    Captured: name, file_path, line_number, end_line, base_classes

    Methods (functions inside a class)

    class AuthService:
    def authenticate(self, username: str, password: str) -> bool:
        ...

    Captured: same as function + class_name

    Imports

    from src.utils.crypto import hash_password
import os

    Captured: module_path, imported_names, is_relative

    Note: import os is captured but filtered out later (standard library = ignored).

    Call Sites

    def authenticate(self, username, password):
    user = find_user(username)         # ← call site
    hashed = hash_password(password)  # ← call site

    Captured: caller_name, callee_name, line_number

    What Gets Skipped

    Skipped Reason
    Standard library imports (import json, import os) Not part of the project; adds noise
    Third-party library calls (requests.get(), np.array()) Only internal connections matter
    Variables and constants Too numerous; not structurally meaningful
    Decorators Stored as properties on functions, not separate nodes
    Type annotations Captured as part of the signature string, not resolved
    Python built-ins (len, print, isinstance, etc.) Filtered via frozenset(dir(builtins))

    Output: FileEntities

    After extraction, each file produces a FileEntities object:

    @dataclass
class FileEntities:
    file_path: str
    functions: list[FunctionEntity]
    classes: list[ClassEntity]
    methods: list[MethodEntity]
    imports: list[ImportEntity]
    calls: list[CallEntity]

    All files are processed, producing a list[FileEntities] that feeds into the graph builder.

    Qualified Names

    To uniquely identify entities across the entire codebase, every entity gets a qualified name:

    Entity Qualified Name Pattern Example
    File file_path src/services/auth_service.py
    Class file_path::ClassName src/services/auth_service.py::AuthService
    Function file_path::function_name src/utils/helpers.py::validate_email
    Method file_path::ClassName.method_name src/services/auth_service.py::AuthService.authenticate

    This prevents ambiguity when two files define entities with the same name (e.g., both utils.py and helpers.py define parse()).

    Source File

    The main entry points are in src/parser/python_parser.py:

    create_parser() -> Parser              # Create the tree-sitter parser
parse_file(path, parser) -> FileEntities    # Parse a single file
parse_directory(path, parser) -> list[FileEntities]  # Parse all .py files

    The extractor logic lives in src/parser/extractors.py. Data models are in src/parser/models.py.
