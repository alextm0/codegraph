    Seed Node Selection

    PPR needs starting nodes (seeds) to determine which part of the code graph to explore. The quality of seeds directly determines the quality of results — bad seeds produce irrelevant context.

    The Problem: Where Do We Start?

    Given a task like "fix the authentication timeout bug," we need to identify the 3–6 most relevant graph nodes to use as PPR seeds.

    Naïve approach: Use all nodes in the current file as seeds.

    Why this fails: - The developer may have been searching for an hour and landed on an unrelated file - The AI agent may be doing multi-step reasoning, not focusing on one file - "Where the cursor is" ≠ "what the task is about"

    The Solution: Multi-Signal Seed Extraction

    Seeds come from three signals with different confidence levels, combined into a single personalization vector.

    Signal 1: Explicit Entity Mentions (Weight: 0.6)

    The AI agent passes a mentioned_entities list when calling the MCP tool. These are entities the agent already identified as relevant from the task description.

    get_relevant_context(
    task_description="fix the authentication timeout bug",
    mentioned_entities=["authenticate", "AuthService", "TokenExpiredError"]
)

    The system looks up each name in the graph. Exact matches get the highest weight (0.6 of the total personalization vector).

    Why highest weight: If the AI agent explicitly names a function, it almost certainly belongs in the seed set.

    Signal 2: BM25 Keyword Search (Weight: 0.3)

    BM25 ranks all graph entities by textual similarity to the task description. The top N results become seeds.

    What is BM25? A text retrieval algorithm. Given a query ("authentication timeout") and a set of documents (all function signatures + docstrings in the graph), it ranks documents by how well they match — considering word frequency, document length, and inverse document frequency (rare words matter more).

    # Corpus: all function signatures + docstrings
corpus = [
    "authenticate(self, username, password) -> bool",
    "validate_token(token: str) -> bool",
    "hash_password(password: str) -> str",
    ...
]

# Query: the task description
query = "fix authentication timeout bug"

# BM25 ranks the corpus
bm25 = BM25Okapi([doc.split() for doc in corpus])
scores = bm25.get_scores(query.split())
top_5 = sorted(zip(scores, corpus))[-5:]

    Why medium weight: BM25 doesn't understand code structure, so it sometimes matches on incidental keywords. It's a useful signal but not as reliable as explicit mentions.

    Signal 3: Current File Hint (Weight: 0.1)

    If the MCP caller provides current_file, all functions/methods in that file are added as weak seeds.

    get_relevant_context(
    task_description="fix timeout",
    current_file="src/services/auth_service.py"
)

    Why lowest weight: The current file is a noisy signal. The developer might be reading unrelated code. But it's not useless — if the task is "continue working on this file," the current-file signal matters.

    Building the Personalization Vector

    All seed signals are combined and normalized to sum to 1.0:

    @dataclass
class SeedNode:
    qualified_name: str   # e.g., "src/services/auth_service.py::AuthService.authenticate"
    weight: float         # raw weight before normalization

@dataclass
class PersonalizationVector:
    seeds: list[SeedNode]

def extract_seeds(
    task_description: str,
    mentioned_entities: list[str] | None,
    current_file: str | None,
    all_entities: list[FileEntities],
    config: dict
) -> PersonalizationVector:
    ...

    Example output:

    PersonalizationVector(seeds=[
    SeedNode("auth_service.py::AuthService.authenticate",  weight=0.50),
    SeedNode("auth_service.py::AuthService",               weight=0.25),
    SeedNode("token_manager.py::validate_token",           weight=0.12),
    SeedNode("crypto.py::hash_password",                   weight=0.08),
    SeedNode("auth_service.py::AuthService.refresh_token", weight=0.05),
])

    Weights sum to 1.0. PPR uses this as the initial probability distribution.

    Why Multi-Signal Matters

    Different scenarios benefit from different signals:

    Scenario Dominant Signal Why
    "Fix bug in authenticate()" Entity match Function name is explicit
    "The login flow is broken" BM25 No specific function; keywords like "login" guide BM25
    "Continue working on auth" Current file Vague task; file context helps
    SWE-bench bug report text Entity match + BM25 Bug reports often name classes; BM25 captures surrounding keywords

    By combining signals, the system degrades gracefully — when one signal is weak, the others compensate.

    Configuration

    # config.yaml
seed_selection:
  entity_match_weight: 0.6   # Weight for explicitly mentioned entities
  bm25_weight: 0.3           # Weight for BM25 top-N matches
  current_file_weight: 0.1   # Weight for entities in the current file
  bm25_top_n: 5              # How many BM25 results to include
