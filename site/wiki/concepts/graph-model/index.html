    Graph Model

    CodeGraph represents a Python codebase as a property graph stored in Neo4j. Every code entity becomes a node; every relationship between entities becomes a directed, weighted edge.

    Node Types

    The graph uses 4 node types, chosen to capture the meaningful structural units in Python code:

    Label Represents Example
    File A .py source file src/services/auth_service.py
    Class A class definition AuthService
    Function A standalone (top-level) function validate_email()
    Method A function defined inside a class AuthService.authenticate()

    Node Properties

    File: 

    file_path   — absolute path on disk
name        — filename only (e.g., "auth_service.py")
line_count  — total lines in the file

    Class: 

    name            — class name (e.g., "AuthService")
qualified_name  — file_path + "::" + name
file_path       — which file this class is in
line_number     — where the class definition starts
end_line        — where it ends

    Function / Method: 

    name            — function name (e.g., "authenticate")
qualified_name  — file_path + "::" + class_name + "." + name  (for methods)
                  file_path + "::" + name                      (for functions)
file_path       — which file this function is in
line_number     — start line
end_line        — end line
signature       — full parameter signature (e.g., "(self, username: str) -> bool")
class_name      — parent class (methods only)

    Node Identity

    Every node is uniquely identified by its qualified_name:

    "src/models/user.py::User"                       ← a Class
"src/services/auth_service.py::AuthService.authenticate"  ← a Method
"src/utils/helpers.py::validate_email"           ← a Function
"src/services/auth_service.py"                   ← a File

    This prevents ambiguity when different files define entities with the same name.

    Edge Types

    The graph uses 4 edge types, each representing a different kind of dependency:

    Type From → To Meaning Example
    CONTAINS File → Class, File → Function, Class → Method Structural containment auth_service.py CONTAINS AuthService
    IMPORTS File → File File-level import dependency auth_service.py IMPORTS user_model.py
    CALLS Function/Method → Function/Method Direct function call authenticate() CALLS hash_password()
    INHERITS_FROM Class → Class Class inheritance AuthService INHERITS_FROM BaseService

    Edge Weights

    Every edge carries a numeric weight property used by PPR during the random walk:

    Edge Type Base Weight Rationale
    INHERITS_FROM 1.0 Strongest coupling — base class changes affect all subclasses
    CALLS 0.8 Direct dependency — callee changes may break caller
    IMPORTS 0.5 File-level connection — weaker than function-level
    CONTAINS 0.4 Structural — relevant but not a functional dependency

    These base weights are then adjusted by in-degree (IDF reweighting) before the GDS projection is created. See Post-Processing →.

    A Worked Example

    Given this code:

    # src/services/auth_service.py
from src.models.user import User
from src.utils.crypto import hash_password

class AuthService(BaseService):
    def authenticate(self, username: str, password: str) -> bool:
        user = User.find_by_username(username)
        return hash_password(password) == user.password_hash

    The graph contains:

    Nodes: 

    (File {name: "auth_service.py"})
(File {name: "user.py"})
(File {name: "crypto.py"})
(Class {name: "AuthService"})
(Class {name: "BaseService"})
(Class {name: "User"})
(Method {name: "authenticate", class_name: "AuthService"})
(Function {name: "hash_password"})
(Method {name: "find_by_username", class_name: "User"})

    Edges: 

    auth_service.py  -[CONTAINS]->    AuthService
auth_service.py  -[IMPORTS]->     user.py
auth_service.py  -[IMPORTS]->     crypto.py
AuthService      -[CONTAINS]->    authenticate
AuthService      -[INHERITS_FROM]-> BaseService
authenticate     -[CALLS]->       find_by_username
authenticate     -[CALLS]->       hash_password

    Querying the Graph

    Find all methods of a class: 

    MATCH (c:Class {name: "AuthService"})-[:CONTAINS]->(m:Method)
RETURN m.name, m.line_number

    Find what a function calls: 

    MATCH (f:Method {name: "authenticate"})-[:CALLS]->(callee)
RETURN callee.name, callee.file_path

    Find all callers of a function: 

    MATCH (caller)-[:CALLS]->(f {name: "hash_password"})
RETURN caller.name, caller.file_path

    Find all classes that inherit from a base: 

    MATCH (child:Class)-[:INHERITS_FROM]->(parent:Class {name: "BaseService"})
RETURN child.name, child.file_path

    See Cypher Queries → for a full reference.

    Schema Design Rationale

    This 4+4 schema follows LocAgent (ACL 2025), which found it captures the vast majority of meaningful structural relationships in Python codebases while keeping the graph manageable. Notably:

    • Variables and constants are not modeled as nodes (too numerous, add noise)
    • Decorators are stored as properties, not edges
    • Docstrings are stored on nodes for BM25 text search in seed selection
    • Standard library and third-party imports are not modeled as IMPORTS edges (only project-internal connections matter)
