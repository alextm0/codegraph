    Personalized PageRank (PPR)

    PPR is the core algorithm of CodeGraph. It scores every node in the code graph by its structural relevance to a given set of starting nodes (seeds). This page explains it from scratch.

    Background: Regular PageRank

    PageRank was invented by Google's founders in 1998 to rank web pages.

    The intuition: Imagine a random web surfer. They start on a random page, click a link, then another, and so on, forever. Pages they visit most often are the most "important" — because important pages have many incoming links from other important pages.

    The teleportation trick: With probability α (typically 0.15), the surfer ignores all links and jumps to a completely random page. This prevents getting stuck in loops and ensures all pages can be reached. The result: every page gets a score (visit frequency). Higher score = more globally important.

    What Makes PPR Different

    In regular PageRank, the teleportation target is any random page with equal probability.

    In Personalized PageRank, teleportation goes to specific pages you choose — called seed nodes. This biases the entire walk toward a particular region of the graph.

    Algorithm Teleports to Answers
    PageRank Any random node "What are the most important nodes globally?"
    PPR Specific seed nodes "What are the most important nodes near my seeds?"

    For code: PageRank tells you the most important functions in the entire codebase. PPR from authenticate() tells you the most important functions relative to authenticate() — the ones most structurally connected to it through call chains, imports, and inheritance.

    How PPR Works Step by Step

    Given a code graph and seed nodes (e.g., AuthService.authenticate):

    1. Initialize. Place all probability mass on the seeds. If there are multiple seeds, distribute proportionally to their weights.

    2. Walk. From each node with probability mass, spread it along edges. Each neighbor gets mass / out_degree. But only spread (1 − α) of the mass; keep α as a pull-back to seeds.

    3. α (alpha) = restart probability = 0.15 by default. Controls how far the walk spreads. Higher α = results stay closer to seeds.

    4. Repeat for 20 iterations (configurable). Scores converge.

    5. Read results. The probability mass at each node after convergence = its PPR score = its structural relevance to the seeds.

    Concrete Example

    Starting PPR from authenticate():

    authenticate()          → 0.150  (high — it's the seed)
hash_password()         → 0.082  (high — directly called by seed)
check_token()           → 0.067  (high — directly called by seed)
User.find_by_username() → 0.045  (medium — called by seed)
BaseService.__init__()  → 0.023  (medium — inherited by seed's class)
process_login()         → 0.018  (medium — calls the seed)
log_error()             → 0.012  (lower — connected but generic)
send_email()            → 0.001  (very low — barely connected)

    The algorithm naturally ranks structurally connected code higher and distant code lower — without any hand-crafted rules.

    Key Parameters

    Parameter Meaning Default Effect of Increasing
    damping_factor (= 1 − α) How far the walk spreads 0.85 Results spread further from seeds
    max_iterations Precision of convergence 20 More precise (diminishing returns after 20)
    tolerance Early stopping threshold 1e-7 Lower = more precise but slower
    top_k How many results PPR returns 20 More results, but more post-processing noise

    In Neo4j GDS: dampingFactor = 1 − α. So dampingFactor = 0.85 means 15% restart probability.

    Why PPR Is the Right Algorithm

    Approach Problem
    1-hop neighbors Misses indirect connections (A→B→C; C is invisible)
    k-hop ego-graph (used by RepoGraph) k=1 too narrow; k=2 explodes exponentially; no ranking within results
    Shortest path Doesn't account for multiple connection paths or connection strength
    Random walk without personalization Returns globally important nodes, not task-relevant ones
    PPR Multi-hop, ranked, personalized, mathematically principled

    PPR's Specific Advantages

    • Multi-hop: Finds indirectly connected but important code (e.g., base class 3 hops away)
    • Ranked: Every node gets a score → take exactly top-k
    • Multiple paths count: Node X reachable via 5 paths scores higher than Y via 1 path
    • Fast: 1–10ms on 50,000-node graphs in Neo4j GDS

    Edge Directionality

    Edges have natural directions (authenticate() CALLS hash_password() is a directed edge). PPR can follow them in different ways:

    Direction What PPR Finds When Useful
    Natural (follow arrows) Things the seed depends on Understanding implementation
    Reverse (follow backward) Things that depend on the seed Impact analysis
    Undirected (both ways) Everything structurally connected General context for AI agents

    Decision: UNDIRECTED. An AI agent fixing authenticate() needs to see hash_password() (to understand the implementation) AND process_login() (to know what calls authenticate() and might break). Undirected is strictly more informative.

    Evaluation includes directed and reversed variants as ablation experiments to measure whether undirected is actually best.

    Running PPR in Neo4j GDS

    The GDS projection is created once per graph; PPR can then run many times with different seeds:

    -- Project the graph into memory (one-time setup)
CALL gds.graph.project(
  'codegraph',
  ['Function', 'Method', 'Class', 'File'],
  {
    CALLS:         {orientation: 'UNDIRECTED', properties: 'weight'},
    IMPORTS:       {orientation: 'UNDIRECTED', properties: 'weight'},
    INHERITS_FROM: {orientation: 'UNDIRECTED', properties: 'weight'},
    CONTAINS:      {orientation: 'UNDIRECTED', properties: 'weight'}
  }
)

-- Run PPR from a seed
MATCH (seed:Method {name: 'authenticate'})
CALL gds.pageRank.stream('codegraph', {
  maxIterations: 20,
  dampingFactor: 0.85,
  sourceNodes: [seed],
  relationshipWeightProperty: 'weight'
})
YIELD nodeId, score
RETURN gds.util.asNode(nodeId).name AS name, score
ORDER BY score DESC
LIMIT 20

    Multiple Seeds with Weights

    PPR accepts multiple seed nodes. The personalization vector distributes probability mass across them:

    # From src/graph/ppr.py
def run_ppr(
    gds: GraphDataScience,
    personalization: PersonalizationVector,  # dict[qualified_name → weight]
    config: PPRConfig
) -> list[PPRResult]:
    ...

    Weights are normalized to sum to 1.0. A seed with weight 0.6 gets 3× the starting probability mass of a seed with weight 0.2.

    Source Files

    File Responsibility
    src/graph/ppr.py PPRConfig, PPRResult, create_gds_client(), project_graph(), run_ppr()
    config.yaml ppr section: damping_factor, max_iterations, tolerance, top_k

    The GDS Python client accesses the underlying driver via gds._driver (an internal API, used deliberately for query execution).
