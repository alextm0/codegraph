    MCP Tools Reference

    CodeGraph exposes three tools over the Model Context Protocol. This page documents each tool's parameters, behavior, and expected output.

    What Is MCP?

    The Model Context Protocol (MCP) is an open standard (Anthropic, November 2024) that lets AI applications communicate with external tools and data sources. It works like a plugin system: any MCP-compatible AI host (Claude Code, Cursor, Claude Desktop) can discover and use MCP servers.

    Transport: CodeGraph uses stdio transport — the MCP server runs as a subprocess of the AI host, communicating over standard input/output. No network configuration needed.

    Entry point: 

    python -m src.mcp_server

    Tool 1: get_relevant_context

    The primary tool. Runs the full pipeline: seed extraction → PPR → post-processing → formatted code snippets.

    Parameters

    Parameter Type Required Description
    task_description string Yes Natural language description of what the AI agent is working on. Used for BM25 seed extraction.
    mentioned_entities list[string] No Entity names the AI agent has explicitly identified as relevant (function/class names). Used for high-weight seed matching.
    current_file string No File path the developer is currently viewing. Used as a low-weight seed hint.
    top_k int No (default: 15) How many results to return. Actual count may be lower due to token budgeting.
    token_budget int No (default: 6000) Maximum tokens of source code to include in the response.

    Example Call

    {
  "tool": "get_relevant_context",
  "arguments": {
    "task_description": "fix the authentication timeout bug",
    "mentioned_entities": ["authenticate", "AuthService", "TokenExpiredError"],
    "current_file": "src/services/auth_service.py",
    "top_k": 15,
    "token_budget": 6000
  }
}

    Return Value

    A JSON string containing a list of context results, ordered by relevance:

    [
  {
    "entity": "AuthService.authenticate",
    "qualified_name": "src/services/auth_service.py::AuthService.authenticate",
    "file": "src/services/auth_service.py",
    "lines": "45-78",
    "relevance_score": 0.1502,
    "source_code": "def authenticate(self, username: str, password: str) -> bool:\n    ..."
  },
  {
    "entity": "TokenManager.validate_token",
    "qualified_name": "src/auth/token_manager.py::TokenManager.validate_token",
    "file": "src/auth/token_manager.py",
    "lines": "12-31",
    "relevance_score": 0.0897,
    "source_code": "def validate_token(self, token: str) -> bool:\n    ..."
  }
]

    When to Use This Tool

    Use this tool at the start of any coding task to get structural context. Pass as specific a description as possible, and include entity names if you've already identified relevant functions.

    Tool 2: query_dependencies

    Returns raw dependency information for a specific entity — no PPR, just direct connections.

    Parameters

    Parameter Type Required Description
    entity_name string Yes Name of the entity to query (function, class, or method name).
    direction string No (default: "both") "upstream" (what calls this), "downstream" (what this calls), or "both".
    depth int No (default: 1) 1 = direct connections only. 2 = include one more hop.

    Example Call

    {
  "tool": "query_dependencies",
  "arguments": {
    "entity_name": "authenticate",
    "direction": "both",
    "depth": 1
  }
}

    Return Value

    {
  "entity": "AuthService.authenticate",
  "file": "src/services/auth_service.py",
  "upstream": [
    {"name": "process_login", "file": "src/controllers/login.py", "relation": "CALLS"}
  ],
  "downstream": [
    {"name": "hash_password", "file": "src/utils/crypto.py", "relation": "CALLS"},
    {"name": "User.find_by_username", "file": "src/models/user.py", "relation": "CALLS"}
  ]
}

    When to Use This Tool

    Use when you need to understand exactly what a specific function calls or what calls it — without the broader PPR exploration. Good for impact analysis ("if I change X, what breaks?") or for navigating a call chain step by step.

    Tool 3: get_graph_stats

    Returns statistics about the code graph. No parameters.

    Example Call

    {
  "tool": "get_graph_stats",
  "arguments": {}
}

    Return Value

    {
  "total_nodes": 1247,
  "node_counts": {
    "File": 87,
    "Class": 234,
    "Function": 412,
    "Method": 514
  },
  "total_edges": 4831,
  "edge_counts": {
    "CONTAINS": 1247,
    "IMPORTS": 312,
    "CALLS": 2891,
    "INHERITS_FROM": 381
  },
  "most_connected_files": [
    {"file": "src/core/base.py", "connections": 89},
    {"file": "src/utils/helpers.py", "connections": 67}
  ],
  "graph_name": "codegraph",
  "projection_exists": true
}

    When to Use This Tool

    Use at the start of a session to understand the codebase size and structure. The most-connected files are often the core modules worth knowing about. The AI agent can also use this to check if the graph has been built before attempting retrieval.

    How Claude Uses These Tools

    A typical Claude Code interaction:

    User: "Fix the authentication timeout bug"

Claude:
1. Calls get_graph_stats → confirms graph exists, sees 1247 nodes
2. Calls get_relevant_context(
       task_description="fix authentication timeout bug",
       mentioned_entities=["authenticate", "timeout", "AuthService"]
   )
3. Receives top 12 code snippets with relevance scores
4. Uses this context to understand the authentication flow
5. Writes the fix

    The AI agent decides when to call the tools and how many times. For complex tasks, it may call get_relevant_context multiple times with different descriptions as it learns more about the problem.

    Source File

    All three tools are defined in src/mcp_server.py using the FastMCP framework:

    from mcp.server.fastmcp import FastMCP

mcp = FastMCP("codegraph")

@mcp.tool()
def get_relevant_context(task_description: str, ...) -> str:
    ...

@mcp.tool()
def query_dependencies(entity_name: str, ...) -> str:
    ...

@mcp.tool()
def get_graph_stats() -> str:
    ...
