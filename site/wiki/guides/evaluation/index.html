    Evaluation Guide

    This guide explains how to evaluate CodeGraph's retrieval quality using the SWE-bench benchmark.

    What We're Evaluating

    The fundamental question: Does PPR retrieve the files that actually need to be changed to fix a bug?

    We answer this with SWE-bench Lite — a dataset of 300 real bug reports where we already know which files were changed in the fix. If PPR's top-k results include those files, it "got it right."

    Important: We are NOT fixing bugs or running an LLM. We are measuring retrieval recall — how well the system identifies where to look.

    The Benchmark: SWE-bench Lite

    SWE-bench Lite provides 300 curated bug instances from popular Python repositories:

    Repository Examples
    Django ORM bugs, middleware issues
    Flask Request handling, routing
    scikit-learn ML algorithm bugs
    matplotlib Plotting/rendering issues
    requests HTTP client bugs

    For each instance: - Repository at a specific commit (before the fix) - Issue text — the GitHub bug report - Gold patch — the actual code changes that fixed the bug (ground truth)

    The evaluation asks: given the issue text, does CodeGraph's top-k output include the files/functions from the gold patch?

    Metrics

    Recall@k

    "Of all files/functions that needed changing, what fraction appear in our top-k results?"

    Recall@k = |gold ∩ top_k| / |gold|

    Example: Bug required changing 3 functions. Top-10 results included 2 of them → Recall@10 = 0.667

    We report Recall@5 and Recall@10. Higher k gives higher recall but requires the AI agent to process more context.

    MRR (Mean Reciprocal Rank)

    "On average, how early does the first correct result appear?"

    MRR = (1/N) × Σ (1 / rank_of_first_correct_result)
    First correct result at rank Contribution
    1 1.000
    3 0.333
    5 0.200
    10 0.100
    Not found 0.000

    Example: Across 3 bug reports, first correct at ranks 1, 3, 7 → MRR = (1.0 + 0.33 + 0.14) / 3 = 0.49

    Higher MRR means correct results appear earlier — the AI agent gets useful context faster.

    Baselines

    Every evaluation needs baselines to establish what "good" looks like.

    Baseline How It Works What It Tests
    Random Pick k random files from the repo Floor — any method must beat this
    BM25 Keyword search of issue text against code contents Text-only retrieval (no graph)
    1-hop ego-graph Return direct graph neighbors of seeds (no PPR) Whether multi-hop adds value over 1-hop

    The goal is for PPR to outperform all three baselines.

    Ablation Studies

    Ablations remove or change one component to measure its contribution.

    Experiment Change What It Reveals
    Remove CALLS edges Graph has only IMPORTS, INHERITS_FROM, CONTAINS Are call relationships important?
    Remove IMPORTS edges Graph has only CALLS, INHERITS_FROM, CONTAINS Are file-level imports important?
    Remove INHERITS_FROM edges Graph has only CALLS, IMPORTS, CONTAINS Does inheritance matter?
    Alpha sweep α ∈ What's the optimal PPR spread?
    Direction sweep Natural vs Reversed vs Undirected PPR Which edge direction is best?
    IDF on/off With vs without IDF edge reweighting Does downweighting utility functions help?
    PPR vs 1-hop PPR vs simple neighbor lookup Is multi-hop necessary?
    Top-k sweep k ∈ How many results are needed for good recall?

    Each ablation is a separate run with exactly one variable changed. The ablations are thesis content regardless of outcome — finding that a component doesn't help is a valid research finding.

    Running the Evaluation

    The evaluation pipeline (evaluations/) is work in progress. The manual process:

    Step 1: Get SWE-bench Lite

    # SWE-bench Lite is available from HuggingFace
pip install datasets
python -c "
from datasets import load_dataset
dataset = load_dataset('princeton-nlp/SWE-bench_Lite', split='test')
print(f'Loaded {len(dataset)} instances')
"

    Step 2: For Each Instance

    for instance in dataset:
    repo = instance['repo']          # e.g., "django/django"
    issue_text = instance['problem_statement']   # bug description
    gold_files = extract_gold_files(instance['patch'])  # files changed in fix

    # 1. Check out repo at the "before fix" commit
    checkout_repo(repo, instance['base_commit'])

    # 2. Build the code graph
    build_graph(repo_path, config)

    # 3. Extract seeds from the issue text
    seeds = extract_seeds(
        task_description=issue_text,
        mentioned_entities=extract_mentioned_names(issue_text),
        config=config['seed_selection']
    )

    # 4. Run PPR
    results = run_ppr(seeds, top_k=10, config=config['ppr'])

    # 5. Compute metrics
    retrieved_files = {r.file_path for r in results}
    recall_at_10 = len(gold_files & retrieved_files) / len(gold_files)
    mrr = compute_mrr(results, gold_files)

    record(instance['instance_id'], recall_at_10, mrr)

    Step 3: Aggregate Results

    avg_recall_5  = mean(results[r]['recall_5'] for r in results)
avg_recall_10 = mean(results[r]['recall_10'] for r in results)
avg_mrr       = mean(results[r]['mrr'] for r in results)

print(f"Recall@5:  {avg_recall_5:.3f}")
print(f"Recall@10: {avg_recall_10:.3f}")
print(f"MRR:       {avg_mrr:.3f}")

    Expected Results

    Based on related work:

    Method Recall@10 (approximate)
    Random ~0.05–0.10
    BM25 ~0.25–0.40
    1-hop ego-graph ~0.30–0.45
    PPR (our method) ~0.40–0.60 (thesis goal)

    These are rough estimates — actual results depend on the specific seed extraction quality and graph completeness.

    Evaluation Directory

    The evaluations/ directory contains: - Scripts for running individual instances - Results from completed runs - Comparison tables across ablation variants

    This directory is not under version control (gitignored) because evaluation results can be large.
