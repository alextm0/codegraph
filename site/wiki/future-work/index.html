    Future Work & Limitations

    This page is an honest assessment of what CodeGraph currently cannot do well, and what would meaningfully improve it.

    Known Limitations

    1. Static Analysis Only

    CodeGraph analyzes code structure at rest — it cannot see dynamic behavior. This means:

    • Monkey-patching is invisible (obj.method = other_function creates a runtime edge that doesn't appear in the graph)
    • eval() and exec() calls are completely unresolvable
    • Dynamic imports (importlib.import_module(config.name)) cannot be followed
    • Duck typing means many method calls remain unresolved without type inference

    Impact: ~5–20% of connections in typical Python codebases are unreachable via static analysis. PPR still works — it just won't traverse those missing edges.

    2. Python Only

    The parser and entity extraction are Python-specific. The architecture is designed to be extensible to other languages (tree-sitter supports 100+), but the current implementation only handles .py files.

    What would be needed for another language: A new language-specific extractor module (replacing extractors.py), handling that language's import syntax and entity types.

    3. Imperfect Import Resolution

    The custom import resolver handles ~80–90% of imports in well-structured Python projects. It fails on:

    • Complex __init__.py re-export chains
    • Namespace packages (PEP 420)
    • Conditional imports (if TYPE_CHECKING: ...)
    • sys.path manipulation at runtime

    Potential fix: Use jedi (a Python type inference library) as a fallback for unresolved imports. This would push resolution close to 100% but adds complexity.

    4. No Incremental Updates

    When code changes, the entire graph must be rebuilt from scratch. A 5–30 second rebuild is fast for evaluation but would be disruptive in a live coding assistant.

    What incremental updates would require: - File change detection (file hashing, filesystem watcher) - Partial graph deletion (remove nodes/edges for changed files) - Partial rebuild (re-parse and re-resolve only changed files) - Projection invalidation and recreation

    5. Utility Function Noise

    IDF reweighting reduces utility function noise but doesn't eliminate it. Functions like validate() or process() that are moderately connected (not enough to be strongly downweighted) may still appear in results when they're not relevant.

    Potential fix: Domain-specific function blocklist, or learned weights from evaluation feedback.

    6. PPR May Not Always Win

    PPR is the thesis's primary contribution, but it's not guaranteed to outperform simpler baselines on all tasks. For tasks that are very localized (one function, obvious name), BM25 alone may be sufficient. For very sparse graphs (under-resolved imports/calls), 1-hop ego-graphs may perform comparably.

    Evaluation ablations will quantify this.

    7. Imperfect Call Resolution

    Without type inference, obj.method() calls where obj is a local variable of unknown type cannot be resolved. This affects method call edges significantly — particularly in code that uses dependency injection or factory patterns.

    Planned Improvements

    Near-Term (Thesis Scope)

    Evaluation completion (Stage 6) Run SWE-bench Lite evaluation across all 300 instances and ablation variants. This is the primary remaining milestone.

    CLI entry point A simple command-line interface to build the graph without writing Python code: 

    codegraph index /path/to/repo
codegraph query "fix the authentication timeout"
codegraph stats

    Better BM25 corpus Currently BM25 searches function signatures + docstrings. Including function bodies (stripped of syntax) would improve recall for tasks where the relevant functions don't have descriptive names.

    Medium-Term (Post-Thesis)

    Incremental graph updates File-change detection + partial graph rebuilding. This would enable real-time CodeGraph in a live coding assistant without the 5–30 second rebuild delay.

    Multi-language support Add JavaScript/TypeScript support (largest potential user base after Python). The graph schema and PPR algorithm are language-agnostic; only the parser and extractor would need new implementations.

    Jedi fallback for import resolution Using jedi as a fallback for the ~15% of imports the custom resolver misses would significantly improve graph completeness for complex repositories.

    Semantic similarity as an additional seed signal Currently seeds are based on exact name matching and BM25 keywords. A semantic embedding model could find entities that are semantically similar to the task description even when terminology doesn't overlap — e.g., finding calculate_sum() when the task mentions "add numbers."

    Long-Term (Research Directions)

    Type-inference-aware call resolution Integrating a lightweight type inference pass (using pyright or mypy's type inference engine) would resolve method calls on typed objects, dramatically improving graph completeness.

    Graph attention for PPR personalization Instead of hand-tuned seed weights (entity_match: 0.6, BM25: 0.3, current_file: 0.1), a learned model could predict optimal personalization vectors from task descriptions and codebase characteristics.

    Cross-repository transfer Building a universal code knowledge graph that spans multiple repositories — useful for understanding library usage patterns across projects.

    Real-time graph streaming As the developer types, update the graph incrementally (at the AST diff level) so that the retrieval system always reflects the current code state.
