    Architecture

    CodeGraph is a pipeline system. Data flows linearly through stages, each producing output consumed by the next. Understanding the architecture means understanding what each stage does, what it produces, and where the code lives.

    High-Level Diagram

    ┌─────────────────────────────────────────────────────┐
│                  Python Repository                   │
│            (source files on disk)                   │
└──────────────────────┬──────────────────────────────┘
                       │
              ┌────────▼────────┐
              │  STAGE 1: PARSE  │
              │  tree-sitter     │   src/parser/python_parser.py
              │  AST per file    │   src/parser/extractors.py
              └────────┬────────┘
                       │  FileEntities per file
              ┌────────▼────────┐
              │ STAGE 2: BUILD   │
              │  GRAPH           │   src/graph/graph_builder.py
              │  Neo4j + GDS     │   src/graph/connection.py
              └────────┬────────┘
                       │  Graph ready in Neo4j
              ┌────────▼────────┐
              │ STAGE 3: QUERY   │
              │  Seed Selection  │   src/retrieval/seed_selection.py
              │  PPR Algorithm   │   src/graph/ppr.py
              │  Post-Processing │   src/retrieval/post_processing.py
              └────────┬────────┘
                       │  Ranked code snippets
              ┌────────▼────────┐
              │  STAGE 4: SERVE  │
              │  MCP Server      │   src/mcp_server.py
              └─────────────────┘
                       │
              ┌────────▼────────┐
              │   AI Agent       │
              │ (Claude, Cursor) │
              └─────────────────┘

    Component Responsibilities

    Stage 1 — Parsing (src/parser/)

    File Responsibility
    python_parser.py Entry point: parse_file(), parse_directory(), create_parser()
    extractors.py Walks AST nodes, extracts functions/classes/methods/imports/calls
    models.py Data models: FileEntities, FunctionEntity, ClassEntity, MethodEntity, ImportEntity, CallEntity
    node_utils.py tree-sitter node traversal helpers

    Input: A path to a .py file or directory Output: list[FileEntities] — one per file, containing all extracted entities

    Stage 2 — Graph Building (src/graph/)

    File Responsibility
    graph_builder.py build_graph() — loads all nodes and edges into Neo4j; clear_database()
    connection.py create_driver(), verify_connectivity(), load_full_config()
    queries.py Read-only graph queries: node lookup, neighbor queries, stats
    ppr.py GDS projection management and PPR execution
    utils.py Shared graph utilities

    Input: list[FileEntities] + Neo4j connection Output: Populated Neo4j database with node and edge weights

    Stage 3 — Retrieval (src/retrieval/)

    File Responsibility
    seed_selection.py extract_seeds() — combines entity match, BM25, and current-file signals into a personalization vector
    post_processing.py apply_idf_weights(), format_context(), count_tokens()
    pipeline.py run_retrieval_pipeline() — orchestrates seed extraction → PPR → post-processing

    Input: Task description string + optional entity hints + optional current file Output: List of ContextResult objects (entity, source code, relevance score)

    Stage 4 — MCP Server (src/mcp_server.py)

    Single file exposing three tools over the Model Context Protocol:

    Tool Purpose
    get_relevant_context Full pipeline: seeds → PPR → formatted code snippets
    query_dependencies Raw graph query — what does entity X call/import?
    get_graph_stats Codebase statistics for the AI agent

    Transport: stdio (local) for Claude Code / Cursor integration Entry point: python -m src.mcp_server

    Data Flow: A Concrete Example

    A developer asks Claude Code: "Fix the authentication timeout bug."

    1. Claude calls: get_relevant_context(
       task_description="fix the authentication timeout bug",
       mentioned_entities=["authenticate", "timeout", "AuthService"]
   )

2. Seed selection:
   - Entity match:  finds AuthService.authenticate in graph  → weight 0.6
   - BM25 search:   finds timeout_handler, token_validator    → weight 0.3
   - Current file:  no file provided                          → skipped

3. Personalization vector:
   { AuthService.authenticate: 0.50, timeout_handler: 0.25, token_validator: 0.25 }

4. PPR runs from these seeds, returns top 20 nodes with scores

5. Post-processing:
   - IDF reweighting reduces generic utility functions (log_error, etc.)
   - Token budgeting reads source code, fills up to 6,000 tokens

6. Result returned to Claude:
   [
     { entity: "AuthService.authenticate", score: 0.15, source_code: "..." },
     { entity: "TokenManager.validate_token", score: 0.09, source_code: "..." },
     ...
   ]

7. Claude uses this context to write the fix

    Key Design Decisions

    Decision Choice Why
    Graph database Neo4j + GDS Built-in PPR, Cypher queries, free for development
    Parser tree-sitter Fast, error-tolerant, industry-standard
    Algorithm Personalized PageRank Multi-hop, ranked, mathematically principled
    Edge direction UNDIRECTED AI agents need both upstream (callers) and downstream (callees) context
    MCP transport stdio Zero-config for local Claude Code / Cursor setup
    Node identity file_path::qualified_name Prevents name collisions across files

    See DECISIONS.md for the full decision log.

    Module Boundaries

    Each module has a single job and can be tested independently:

    parser/     → has no knowledge of Neo4j
graph/      → has no knowledge of retrieval
retrieval/  → has no knowledge of MCP
mcp_server  → thin orchestration layer, no business logic

    Dependencies flow in one direction: mcp_server → retrieval → graph + parser.
