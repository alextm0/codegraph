    Import & Call Resolution

    This is the trickiest part of the pipeline. tree-sitter gives us raw entities per file, but it doesn't connect them across files. Resolution fills in those cross-file connections.

    Import Resolution

    The Problem

    tree-sitter parses from src.utils.helpers import validate_email and gives us the string "src.utils.helpers" and the name "validate_email". But tree-sitter does not tell us which actual file on disk "src.utils.helpers" refers to.

    Import resolution turns "src.utils.helpers" → /project/src/utils/helpers.py.

    Without this, we can't create IMPORTS edges in the graph because we don't know which files are connected.

    How Python Imports Work (Simplified)

    When Python sees from src.utils.helpers import validate_email:

    1. Take the dotted path src.utils.helpers
    2. Convert dots to directory separators: src/utils/helpers
    3. Look for either:
    4. A file: src/utils/helpers.py
    5. A package: src/utils/helpers/__init__.py
    6. Inside that file, look for validate_email

    Relative imports start with dots: from ..models import User. Each dot means "go up one directory." In src/services/auth_service.py, ..models means src/models.

    Difficulty Categories

    Easy Cases (~80% of imports in well-structured projects)

    from src.utils.helpers import validate_email
# → Find src/utils/helpers.py in the project. Done.

    Straightforward string manipulation + filesystem check.

    Medium Cases (~15%)

    Relative imports: 

    from ..models import User
# → Need current file's location, count dots, go up directories,
#   find models.py or models/__init__.py

    Re-exports through __init__.py: 

    # src/models/__init__.py contains:
from .user import User
from .post import Post

# So "from src.models import User" actually resolves to src/models/user.py

    For the initial implementation, resolving to the __init__.py file itself is acceptable — the connection between src/models/ and the importing file is still captured.

    Impossible Cases (~5% — skipped)

    module = importlib.import_module(config.module_name)   # runtime string
exec("from " + module_name + " import something")      # generated code

    These depend on values only known at runtime. No static analysis tool can resolve them. We log them as unresolved and move on.

    The Resolver

    A custom resolver (~150–200 lines of Python) handles easy and medium cases:

    def resolve_import(import_path: str, current_file: str, project_root: str) -> str | None:
    if import_path.startswith('.'):
        # Relative import: count leading dots, resolve relative to current file
        dots = len(import_path) - len(import_path.lstrip('.'))
        remaining = import_path.lstrip('.')
        base_dir = current_file_directory
        for _ in range(dots):
            base_dir = parent(base_dir)
        relative_path = remaining.replace('.', '/')
    else:
        # Absolute import: resolve from project root
        relative_path = import_path.replace('.', '/')
        base_dir = project_root

    # Try as a module file
    candidate = base_dir / (relative_path + '.py')
    if exists(candidate):
        return candidate

    # Try as a package
    candidate = base_dir / relative_path / '__init__.py'
    if exists(candidate):
        return candidate

    return None  # Unresolved — log it and continue

    Why custom over jedi: The custom resolver is educational (good for thesis) and sufficient for well-structured SWE-bench repos (Flask, Django, scikit-learn). If resolution falls below ~75%, jedi can be added as a fallback — it understands Python's full import system.

    Reporting: The import resolution rate (e.g., "resolved 87% of 610 imports in Flask") is included in the thesis as an honesty metric.

    Call Resolution

    The Problem

    Entity extraction says: "function authenticate calls something named hash_password." But which hash_password? There might be multiple definitions with that name across the codebase.

    Call resolution connects call sites (where a function is called) to function definitions (where it's defined).

    How It Works

    Call resolution uses the information from import resolution:

    # In auth_service.py:
from src.utils.crypto import hash_password    # ← import resolution → src/utils/crypto.py

def authenticate(self, username, password):
    hashed = hash_password(password)           # ← call resolves to src.utils.crypto.hash_password

    The resolution logic — when we see a call to some_function() inside a file:

    Check Resolution
    Was some_function imported from another file? Resolve to that file's definition
    Is some_function defined in the same file? Resolve to the local definition
    Is it a self.method() call? Resolve to the method on the same class
    Can't determine? Log as unresolved

    For method calls on objects (user.save()), the challenge is knowing the type of user. Heuristics used: - If user was assigned from User.find_by_id(...) → guess User instance - If type hints are present (user: User = ...) → use the type hint - Otherwise → log as unresolved

    What Gets Created

    Each resolved call becomes a CALLS edge in the graph:

    authenticate() ──CALLS──▶ hash_password()
authenticate() ──CALLS──▶ User.find_by_username()
process_login() ──CALLS──▶ authenticate()

    Expected resolution rates: For Python projects with good structure and type hints, expect 60–80% resolution. This is acceptable — PPR works with incomplete graphs. It just won't find connections through the missing edges.
